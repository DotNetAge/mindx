# 运行时编排：AI Agent 系统的另一种架构方式

## 所有 Agent 平台都做错的一件事

每个 AI Agent 平台都在问同一个问题："如何让多个 Agent 协作完成一个复杂任务？"

答案出奇一致：

- **AutoGen**：代码定义 agent 间的对话图
- **CrewAI**：YAML 定义团队和工作流
- **LangGraph**：用图 DSL 定义状态机
- **Semantic Kernel**：用 C# 代码编排 planner
- **OpenAI Assistants**：硬编码的 function calling chain

共同点：**编排发生在开发时，不在运行时。**

开发者要预先想好所有可能的流程，写成图、链、或 YAML。LLM 只是这个预定义流程里的一个执行节点。

这就像用流程图来管理一家公司——你画好了部门架构、汇报关系、审批流程，然后让员工照着走。员工不需要思考，因为你想好了。但用户的需求从来不会按照你画的流程图来。

## 另一种架构：Skill 作为原子能力，LLM 作为运行时编排器

如果把架构反过来呢？

不需要预先定义流程。只需要定义**能力模块**（我们叫它 Skill），然后让 LLM 根据用户的输入，在运行时自己决定怎么组合它们。

每个 Skill 是一份纯文本的 Markdown 文件，告诉 LLM：

- 这个能力什么时候该用
- 使用它的工作流程是什么
- 有哪些脚本可以调用

没有代码、没有图、没有 YAML。就是一篇文章。

LLM 通过一个普通的工具调用加载它：

```
Skill("project-manager")
```

然后这篇文章的内容就被追加到对话上下文中。LLM 现在知道了项目管理的全部指令。

关键在于：**Skill 文件里可以写"如果需要，加载另一个 Skill"。**

```
project-manager 的指令里有这么一段：
  "如果需要寻找或创建专家 Agent，加载 find-experts skill"
```

所以当 PM 在分解目标时发现需要找一个设计师，它可以：

```
Skill("find-experts")
```

find-experts 的指令也被追加进来。LLM 同时看到两个 Skill 的指令，可以在它们之间自由切换。

这不是代码层面的组合——这是**文本层面的组合**。两个 Skill 的文本在同一个对话上下文里共存，LLM 自己决定什么时候用哪个。

## 为什么这个模式有效

因为 LLM 最擅长的就是处理文本。

你把 project-manager 的指令给它，它知道怎么当 PM。你把 find-experts 的指令也给它，它还知道怎么找专家。这两份指令同时存在时，LLM 不会混淆——它能理解"我现在是在做项目管理，但需要帮助时可以去找专家"。

这不靠框架支持。**SkillTool 就是一个普通的文件读取工具**，读谁的文件都行，读几次都行。组合发生在 LLM 的注意力机制里，不在代码的类继承里。

## 这意味着什么

**1. 开发者不需要预先编排流程**

只需要写 Skill 文档。每个 Skill 是独立的能力单元，不依赖其他 Skill。组合是 LLM 在运行时决定的。

**2. Skill 之间可以自然嵌套**

```
用户：帮我运营小红书
  ↓ LLM 加载 project-manager
  ↓ 分解目标时发现缺设计师
  ↓ LLM 加载 find-experts → 创建 @designer → 回到 project-manager 继续安排
```

整个过程中，LLM 在多个 Skill 之间自由切换，开发者不需要写一行编排代码。

**3. Skill 的边界是语义的，不是代码的**

两个 Skill 不需要定义接口、不需要共享类型、不需要版本对齐。它们只需要用同一种语言（自然语言）书写。只要 LLM 能理解，它们就能协作。

**4. 这跟 OPC 不冲突**

实际上，OPC 是这个模式的一个具体应用——project-manager 是 PM 的能力，find-experts 是 HR 的能力，AgentTalk 是通信工具。把它们组合起来，就是一个公司。

## 更深一层：SOP 嵌套 SOP

这个模式真正强大之处在于：**Skill 本身就是一份完整的 SOP（标准作业流程）**。它不是工具描述，不是 API 文档——它是一份教 LLM 怎么做完整一件事的操作手册。

当 Skill 可以嵌套 Skill，就意味着 SOP 可以嵌套 SOP，最终实现现实世界中真实业务流程的完整表达：

```
project-manager（项目管理的 SOP）
  ├── Phase 1: 沟通量化目标
  ├── Phase 2: 分解安排工作
  │   └── 如果缺专家 → 加载 find-experts（招聘的 SOP）
  │       ├── 列出所有 Agent
  │       ├── 如果没有合适的 → 创建新 Agent
  │       └── 分配任务
  ├── Phase 3: 跟踪进度
  │   └── 收到 AgentTalk 汇报 → 评估 → 调整
  └── Phase 4: 汇报用户
```

每一层 SOP 都是完整的、自洽的、可独立使用的。嵌套不是代码级的 import——是文本级的"加载到上下文"。

### 跟工具嵌套的区别

工具也可以嵌套（比如 TaskCreate → SubAgent → CollectResults 的正交组合），但那是在**框架层**——工具是平台内置的原子能力，代码在 goreact 内部。

Skill 嵌套是在**应用层**——Skill 是框架外的文档，它不依赖框架，只依赖脚本、Python、外部 API、或者任何系统上有的程序。

```
框架层（GoReact）：工具嵌套
  TaskCreate → SubAgent → CollectResults
  平台级，代码定义，硬编码在框架内

应用层（MindX Skill）：Skill 嵌套
  project-manager → find-experts → create_agent.py
  应用级，文档定义，依赖外部脚本
```

工具嵌套是**平台能力**的组合，Skill 嵌套是**业务能力**的组合。前者由框架开发者定义，后者由业务使用者定义。

这意味着 Skill 的生态可以不依赖框架更新而独立成长——写一个新的 Skill 就是写一份 Markdown 文件，不涉及框架代码、不需要 PR 合并、不需要版本发布。

### 跟"工具即编排"的本质区别

现有平台走的是 **工具即编排** 的路线：

```
系统启动时 → 加载 50 个工具定义到 system prompt
           → 每个工具定义约 200 tokens
           → 固定消耗 ~10K tokens
           → LLM 在 50 个工具中选 2-3 个用
```

无论用户说什么，这 50 个工具定义都在那里，占着 token 预算。无关的工具也加载。

Skill 即编排完全不同：

```
用户说"帮我运营小红书"
  → 加载 project-manager（~2K tokens）
  → 分解时发现缺设计师
      → 加载 find-experts（~1K tokens）
      → 创建 @designer
      → find-experts 不再需要，自然移出上下文
  → 继续 project-manager 流程
```

**Token 消耗与实际工作流成正比**，不是与系统的总能力成正比。

| | 工具即编排 | Skill 即编排 |
|---|---|---|
| 加载策略 | 全部预加载 | **按需懒加载** |
| Token 开销 | 固定高位（~10K+） | **与实际流程正相关** |
| 组合能力 | 函数级（参数传递） | **文档级（上下文共存）** |
| 流程表达 | DAG / 状态机 | **SOP 嵌套 SOP** |
| 开发者体验 | 写代码、定义接口 | **写 Markdown、引用其他 Skill** |
| 运行时灵活性 | 预定义，不可变 | **LLM 自主决定路径** |

### 现实意义

这意味着一家公司的全部 SOP 可以被写成一堆 Markdown 文件。每个部门有自己的 Skill，每个 Skill 可以引用其他 Skill。LLM 作为"员工"，按需加载、按需组合。

入职培训不再是看文档——是加载对应的 Skill。

跨部门协作不再是调 API——是加载对方的 Skill。

一个 Skill 就是一份标准作业流程。一堆 Skill 就是一家公司。

## 跟现有方式的对比

| | 现有平台 | 这种架构 |
|---|---|---|
| 编排时机 | 开发时（代码/YAML/图） | **运行时**（LLM 自主决定） |
| 能力单元 | 函数/工具/插件 | **自然语言文档** |
| 组合方式 | 代码定义接口 | **文本在上下文中共存** |
| 扩展方式 | 写代码、PR、部署 | **写 Markdown 文件** |
| 组合深度 | 受限于 API 边界 | **受限于 LLM 的理解力** |

## 局限

这种模式依赖于 LLM 的上下文窗口——每个 Skill 都会占用 token。如果 LLM 一次加载了太多 Skill，上下文会被撑满。

但这是一个正在快速解决的技术问题——从 GPT-4 的 8K 到今天的 128K、200K，上下文窗口的增长曲线决定了这个限制是暂时的。

另一个问题是：LLM 是否能正确判断什么时候该加载什么 Skill。这取决于 Skill 描述的质量。写得好的 Skill（像 project-manager 那样说清楚"什么时候用"）LLM 基本不会选错。

## 结论

这种架构方式不是某个人的发明，而是 LLM 能力发展到一定阶段后的自然产物。当模型足够强大，自然语言就能作为编排语言——不需要 DSL，不需要图，不需要 YAML。

Skill 就是能力，对话上下文就是总线，LLM 就是运行时编排器。

没有比这更简单的架构了。
