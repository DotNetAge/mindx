# MindX TUI Architecture V2 — 纯 ELM 组件化设计

> 本文件是架构设计文档，不含代码实现细节。
> UML 是唯一的架构语言。

---

## 1. 设计原则 — 严格遵循 Bubble Tea MVU / ELM 架构

> **核心信条：整个 UI 只能有一种模型（tea.Model），禁止以任何借口（性能、灵活等）破坏 MVU 设计模式。**
> 组件不是独立的 tea.Model，而是 rootModel 的内部状态分段，通过统一的 Update(Msg) 接口变更。

```
┌──────────────────────────────────────────────────┐
│                 七条核心原则                       │
├──────────────────────────────────────────────────┤
│                                                   │
│  ① 单一 Model 原则                                │
│     rootModel 是整个 TUI 中唯一的 tea.Model       │
│     子组件是普通 struct，不是 tea.Model            │
│     禁止子组件拥有自己的 tea.Program               │
│                                                   │
│  ② 数据与行为分离                                 │
│     Data = 纯 struct（无方法 / 仅只读 getter）    │
│     Behavior = Update(Data, Msg) → (NewData, Cmd) │
│                                                   │
│  ③ 唯一状态变更入口                               │
│     所有状态修改必须经过 Update 函数               │
│     禁止 setter，禁止外部直接写字段                │
│     禁止从 goroutine 直接调组件方法                │
│                                                   │
│  ④ 组件契约统一                                   │
│     每个子组件实现同一接口：                       │
│     Update(Msg) → (Self, Cmd)                     │
│     View() → string                               │
│                                                   │
│  ⑤ 无 exception 原则                              │
│     任何情况下不绕过上述原则                      │
│     不存在"这里性能特殊所以用 setter"的例外        │
│     如果发现模式被打破，修复模式而非添加例外        │
│                                                   │
│  ⑥ 设计即代码                                     │
│     代码必须严格按照本设计文档实现                │
│     §1-§12 中的每一条规则都对应具体的代码约束      │
│     不允许出现"设计这样写但我换个写法"的情况      │
│     如果代码无法匹配设计，修改代码而非修改设计      │
│                                                   │
│  ⑦ 优先内置组件                                   │
│     界面优先采用 Bubble Tea / Bubbles 提供的组件   │
│     例如：textarea、viewport、list、table 等        │
│     在未找到对应内置组件时才选择手写实现            │
│     避免重复造轮子，保持一致性                       │
└──────────────────────────────────────────────────┘
```

```
│  ┌──────────────────┐  ┌──────────────┐                     │
│  │ NotificationBar  │  │ ChoicesPanel │                     │
│  │  (浮动型)        │  │  (模态型)    │                     │
```

## 4. 消息层 — 强类型消息

所有事件分为三类，每类下细分具体消息类型。
不再有任何 `string contentType` 的 switch。

### 5.2 区块划分

版面分为三个自然区块，区块之间以空行分隔：

| 区块           | 视觉标识                           | 说明                   |
| -------------- | ---------------------------------- | ---------------------- |
| **思考区**     | 用户输入 → 第一个 ActionStart 之间 | 流式思考内容，直通显示 |
| **工具调用区** | ActionStart → FinalAnswer 之间     | 工具执行记录，支持折叠 |
| **最终回答区** | FinalAnswer 及之后                 | 只返回一个最终结果     |

> 注意：区块划分是**自然区隔**，不以 `---思考区---` 等标注行作为标题。
> 区块之间用空行隔开，用户通过图标和上下文自然区分。

### 5.4 区块详情 — 思考区

```
布局:
  ● 深度思考中                    ← 等待第一个 Delta 时
  ● 用户让我总结之前的搜索结果，  ← Delta 到达后实时替换等待文字
     我要分析电商行业的影响...     ← 内容直通，不做任何处理
     … +N lines (ctrl+o to expand)← 默认折叠，仅显示最近三行
     [Tokens: 1,234 in / 5,678 out]
  ● 第二轮思考的内容              ← 多轮：依次追加
     … +N lines (ctrl+o to expand)
     [Tokens: 789 in / 4,321 out]
  ● 第三轮...                     ← 正在流式输出的当前轮次
  ● 仍继续...

规则:
  • 内容 100% 流式直通：Agent 输出什么就显示什么
  • 不做 JSON 解析，不做格式化（如 "reasoning"/"decision" 等字段原文显示）
  • 多轮思考：每轮完整保留，按顺序追加显示
  • 等待第一个 Delta 时显示 "深度思考中"，图标白蓝闪烁
  • Delta 到达后：内容替换等待文字，图标继续闪烁
  • 思考完成（ThinkingDone）后：图标停止闪烁，内容保留
  • 思考内容以斜体灰色显示（Italic, #888888）
  • 思考区结束于第一个 ActionStart 事件
  • 默认折叠：每轮思考内容默认仅显示最近三行
  • 超过三行的部分以 "… +N lines (ctrl+o to expand)" 省略
  • Ctrl+O 切换展开/折叠：展开后显示该轮全部内容
  • 展开/折叠状态由 AnswerData.ThinkingCollapsed 控制
  • 当前正在流式输出的轮次（isThinking=true）始终完全展开
  • 历史已完成的轮次（ThinkingRound）默认折叠
  • 每轮思考完成后，在内容底部追加一行 Token 统计：
    `[Tokens: {TokensIn} in / {TokensOut} out]`
    数字带千分位分隔，颜色为暗灰色（#666666），不加粗

状态机:
  WaitFirstDelta ──ThinkingDelta──→ Streaming (当前轮完全展开)
  Streaming ──ThinkingDone──→ RoundDone (归档后变为折叠)
  RoundDone ──ThinkingDelta──→ Streaming (下一轮，完全展开)
  RoundDone ──ActionStart──→ End (思考区关闭)
```

### 5.5 区块详情 — 工具调用区

```
布局:

    ⎿ 完成 (N lines)                                 ← 折叠状态
  ⏺ ToolName(param: value) | 预计消耗 N Tokens     ← 完成

      … +N lines (ctrl+o to expand)

规则:
  • 工具按执行顺序排列，每个工具占一行（含参数预览）
  • 执行中：图标白绿闪烁，显示参数和 Token 预估
  • 完成：图标固定绿色，默认折叠输出结果
  • 失败：图标红色，显示失败原因
  • 折叠状态显示 "完成 (N lines)" 或 "失败: 错误信息"
  • 展开状态显示前三行内容 + "… +N lines (ctrl+o to expand)"
  • Ctrl+O 切换折叠/展开
  • 展开/折叠状态记录在 ActionStep.Collapsed 中
  • 工具区结束于 FinalAnswer 事件

格式:
  ⏺ToolName(param_str) | 预计消耗 N Tokens | ProgressText     ← 执行中
  ⏺ToolName | 预计消耗 N Tokens | [+] Show output             ← 完成折叠
```

### 5.9 InputArea 与建议区渲染规范

InputBox 有**多种渲染状态**：普通输入模式 → 命令建议模式 → Agent 建议模式 → 模型建议模式 → 会话建议模式。
普通输入模式始终存在，建议模式在输入特定前缀时触发并在输入区下方追加建议面板。

---

#### 5.9.0 建议面板通用交互规范（焦点管理）

⚠️ **核心原则：建议面板是「非模态」的辅助输入工具，不抢占输入焦点**

##### 焦点状态机

```
┌─────────────────────────────────────────────────────────────┐
│                    焦点状态转换图                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   [Input Focus] ──→ [Suggestion Active]                    │
│        ↑                  │                                │
│        │    ┌─────────────┼──────────────┐                 │
│        │    ↓             ↓              ↓                 │
│        │  [↑↓导航]     [Tab补全]     [其他键]              │
│        │    │             │              │                 │
│        │    │         [退出建议]     [回退到Input]          │
│        │    │             │              │                 │
│        └────┴─────────────┴──────────────┘                 │
│                                                             │
│   规则：                                                    │
│   • ↑↓ 键：在建议列表中移动选择（保持 Suggestion Active）   │
│   • Tab 键：补全选中项，退出建议（回到 Input Focus）        │
│   • Enter 键：执行/发送（根据上下文决定行为）               │
│   • 其他可打印字符：立即回退到 Input Focus 并插入字符       │
│   • Esc 键：清除建议，保持当前输入                          │
│   • Backspace：删除字符 + 更新过滤条件                      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

##### 键盘事件处理优先级

| 优先级 | 按键类型     | 处理方式         | 行为                                 |
| ------ | ------------ | ---------------- | ------------------------------------ |
| **P0** | `↑` `↓`      | **建议面板拦截** | 移动选择项，不传入输入框             |
| **P1** | `Tab`        | **建议面板拦截** | 执行补全，退出建议模式               |
| **P2** | `Enter`      | **上下文相关**   | 见各子章节的具体行为                 |
| **P3** | `Esc`        | **建议面板拦截** | 清除建议面板                         |
| **P4** | `Backspace`  | **双向处理**     | 输入框删除 + 更新过滤                |
| **P5** | **其他字符** | **回退到输入框** | 插入字符，保持建议（如果仍匹配前缀） |

##### 实现要求

```go
// 伪代码示例：handleKey() 的实现逻辑
func (i *InputArea) handleKey(k KeyPressMsg) {
    // 1. 检查是否有活跃的建议面板
    if i.hasActiveSuggestion() {
        switch k.String() {
        case "up", "down":
            i.suggestion.Navigate(k)      // P0: 导航
            return                         // 不传入输入框
        case "tab":
            i.suggestion.Complete()       // P1: 补全
            return                         // 退出建议模式
        case "esc":
            i.clearSuggestion()           // P3: 退出建议面板（保留输入内容）
            return                         // ⚠️ 绝不退出程序！
        }
        
        // P5: 其他字符 → 回退到输入框
        if isPrintable(k) {
            i.insertChar(k)               // 插入字符
            i.updateSuggestion()          // 重新计算建议
            return
        }
    }
    
    // 无建议面板时的默认处理...
    // ⚠️ 特殊处理：Esc 键在无建议面板时的行为
    case "esc":
        i.TextBuffer.Reset()             // 清空输入框
        i.CursorPos = 0                  // 重置光标位置
        return                           // ⚠️ 仅清空输入，绝不退出程序！
}
```

##### Esc 键的双模行为

⚠️ **Esc 键的行为取决于当前状态，且绝不用于退出程序**

| 当前状态                    | Esc 键行为   | 效果                                     |
| --------------------------- | ------------ | ---------------------------------------- |
| **有活跃的建议面板**        | 退出建议面板 | 清除建议面板显示，**保留**输入框中的文本 |
| **无建议面板 + 输入框非空** | 清空输入框   | 删除所有已输入的字符，光标归位到起始位置 |
| **无建议面板 + 输入框为空** | 无操作       | 忽略按键（或可选择性发出提示音）         |

**❌ 绝对禁止的行为：**
- ❌ 使用 Esc 退出程序（只能通过 `Ctrl+C`）
- ❌ 在有建议面板时同时清空输入框和面板

**设计理由：**
1. **防止误操作**：用户可能误触 Esc，不应导致程序退出
2. **符合终端惯例**：大多数现代 TUI 应用（如 vim、htop）使用 Esc 返回上一级或取消操作
3. **一致性保证**：与 `Ctrl+C` 的语义分离——Ctrl+C = 强制终止，Esc = 轻量取消

##### 一致性约束

所有建议面板类型（CommandSuggestion、AgentSuggestion、ModelSuggestion、SessionSuggestion）**必须遵循**：

1. ✅ **统一的视觉反馈**
   - 当前选中项使用 `>` 前缀 + 高亮色
   - 非选中项左对齐，无前缀
   
2. ✅ **统一的导航语义**
   - `↑` 向上移动（SelIdx--）
   - `↓` 向下移动（SelIdx++）
   - 循环边界：第一项向上 → 跳到末尾；最后一项向下 → 跳到首项

3. ✅ **统一的过滤机制**
   - 实时过滤：每输入一个字符立即重新筛选
   - 子串匹配：支持任意位置的子串（如 `/in` 匹配 `/init`）
   - 大小写敏感：按配置决定（推荐大小写不敏感）

4. ✅ **统一的状态清理**
   - 补全后必须清除建议面板
   - 执行后必须清空输入框并清除建议
   - 切换前缀类型时（如 `/` → `@`）必须重置状态

#### 5.9.1.1 命令建议交互行为

⚠️ **重要：命令建议支持两种交互模式——「补全」和「执行」**

##### 模式一：Tab 键 — 指令补全（Completion）

**触发条件**：用户在命令建议模式下按下 `Tab` 键

**行为流程**：
```
用户输入: "/in"                    用户按下 Tab
    ↓                              ↓
┌─────────────────────┐      ┌─────────────────────┐
│  ❯ /in              │      │ ❯ /init █           │
│  ───────────────────│  →   │  ───────────────────│
│  > /init  Initialize│      │                      │
│    /info  Show info │      │  (建议面板已清除)     │
└─────────────────────┘      └─────────────────────┘
```

**具体操作**：
  1. 获取当前高亮的命令项 (`cmdSuggest.SelIdx`)
  2. **清空输入框**
  3. 将选中命令补全到输入框：`/{CommandName} `
  4. 光标移至末尾
  5. **清除建议面板**，回到 NormalInput 模式
  6. 用户可继续输入参数，然后按 Enter 执行

**适用场景**：
  - 命令需要参数（如 `/agents list`）
  - 用户想确认命令名称后再执行

---

##### 模式二：Enter 键 — 直接执行（Execution）

**触发条件**：用户在命令建议模式下按下 `Enter` 键

**行为流程**：
```
用户输入: "/init"                   用户按下 Enter
    ↓                               ↓
┌─────────────────────┐       ┌─────────────────────┐
│  ❯ /init            │       │  (输入框清空)        │
│  ───────────────────│  →    │  ───────────────────│
│  > /init  Initialize│       │                      │
└─────────────────────┘       └─────────────────────┘
                                   ↓
                          发送 SlashCommandMsg
                          { Name: "init", Args: [] }
                                   ↓
                           rootModel.handleSlashCommand()
                                   ↓
                            执行命令并显示结果
```

**具体操作**：
  1. 解析输入文本：提取命令名和参数
  2. **清空输入框**
  3. 发送 `SlashCommandMsg{Name, Args}` 消息
  4. **清除建议面板**
  5. `rootModel` 接收消息并调用 `handleSlashCommand()`
  6. 命令注册表查找对应命令并执行
  7. 结果通过 `NotificationBar` 显示

**适用场景**：
  - 无参命令（如 `/clear`, `/help`）
  - 用户确定要立即执行

| `↑` (Up)    | 上移选择 | `SelIdx--`，循环到最后一项     |

### 6.2 /doctor — 系统诊断指令

### 6.3 /model — 模型切换指令

#### 功能描述

用于永久性更改当前 Agent 的模型配置。修改的是 **Agent 定义文件**（`~/.mindx/agents/{agent_name}.yml`），而非运行时状态。

⚠️ **核心设计原则：**
- Agent 在完成完整的 TAO（Think-Act-Observe）循环前不接受任何变更
- Agent 完成循环后会自动销毁，下次对话时从定义文件重新加载
- 因此 `/model` 只需修改磁盘上的定义文件，无需重建内存中的 Agent 实例

#### 语法格式

```
/model <model_name>
```

**参数**：
- `model_name` (必需): 目标模型的名称（如 `claude-sonnet-4-20250514`）

**示例**：
```bash
/model claude-opus-4-20250514    # 切换到 Claude Opus
/model gpt-4o                    # 切换到 GPT-4o
```

#### 触发条件与建议面板

当用户输入 `/model` 时：
1. **立即激活 ModelSuggestion 面板**（见 §5.9.3）
2. **列出所有可用模型**：从 `App.Models().List()` 获取
3. **支持缩写过滤**：输入 `/mod cl` 只显示包含 "cl" 的模型

#### 行为流程

##### 场景 A：通过建议面板选择（推荐）

```
1. 用户输入: "/model"
   ↓
2. 显示 ModelSuggestion 面板（列出所有可用模型）
   ↓
3. 用户按 ↑↓ 选择 "claude-sonnet-4-20250514"
   ↓
4. 用户按 Enter
   ↓
5. 发送 SlashCommandMsg { Name: "model", Args: ["claude-sonnet-4-20250514"] }
   ↓
6. rootModel.handleSlashCommand()
   ↓
7. 调用 app.SwitchAgentModel(currentAgentName, "claude-sonnet-4-20250514")
   ├── 从 Agents().Get(agentName) 获取 Agent 配置对象
   ├── 修改配置对象的 Model 字段为新值
   ├── 将更新后的配置写回定义文件 (~/.mindx/agents/{name}.yml)
   └── 清空 agentCache[agentName] 缓存（强制下次重新加载）
   ↓
8. StatusBar 显示: "✅ 已更新 {agent} 模型为: claude-sonnet-4-20250514"
   └── 提示: "下次对话时生效"
```

##### 场景 B：直接输入完整命令

```
1. 用户输入: "/model gpt-4o" + Enter
   ↓
2. 发送 SlashCommandMsg { Name: "model", Args: ["gpt-4o"] }
   ↓
3. 同场景 A 的步骤 6-8
```

#### 关键实现细节

**为什么不需要重建 Agent？**

```go
// Agent 生命周期说明：
//
// 1. 用户发送消息
//    ↓
// 2. app.ResolveAgent(name) 或 app.getMaster()
//    ├── 检查 agentCache 命中？
//    │   ├── 命中 → 返回缓存实例（复用）
//    │   └── 未命中 → 从定义文件加载 → 创建新实例 → 缓存
//    ↓
// 3. agent.Ask(sessionID, message)
//    ↓
// 4. Agent 执行 TAO 循环 (Think → Act → Observe)
//    ↓
// 5. 循环完成 → 返回 FinalAnswer
//    ↓
// 6. 下次调用时重复步骤 2（如果缓存被清除则重新加载）
//

// 因此 SwitchModel 只需：
func (app *App) SwitchAgentModel(agentName, modelName string) error {
    // 1. 获取 Agent 配置（从注册表，非内存实例）
    cfg := app.Agents().Get(agentName)
    if cfg == nil {
        return fmt.Errorf("agent %q 不存在", agentName)
    }

    // 2. 验证目标模型可用
    model := app.Models().Get(modelName)
    if model == nil || !model.Enabled {
        return fmt.Errorf("模型 %q 不可用", modelName)
    }

    // 3. 修改配置对象的 Model 字段
    oldModel := cfg.Model
    cfg.Model = modelName

    // 4. 写回定义文件（永久性变更）
    agentFile := filepath.Join(app.settings.AgentsDir(), agentName+".yml")
    if err := writeAgentConfig(agentFile, cfg); err != nil {
        // 回滚：恢复旧值
        cfg.Model = oldModel
        return fmt.Errorf("保存失败: %w", err)
    }

    // 5. 清除该 Agent 的缓存（强制下次重建）
    app.agentMu.Lock()
    delete(app.agentCache, agentName)
    app.agentMu.Unlock()

    // 6. 如果当前 master 就是被修改的 Agent，也清除 master 缓存
    if app.currentAgentName() == agentName {
        app.masterMu.Lock()
        app.master = nil
        app.masterMu.Unlock()
    }

    app.logger.Info("agent model updated",
        "agent", agentName,
        "old_model", oldModel,
        "new_model", modelName,
    )

    return nil
}
```

#### 错误处理

| 错误情况     | 处理方式                               | 用户反馈                                               |
| ------------ | -------------------------------------- | ------------------------------------------------------ |
| 模型不存在   | 检查 `Models().Get(name)` 是否返回 nil | NotificationBar 显示错误：`"模型 'xxx' 不存在"`        |
| 模型未启用   | 检查 `model.Enabled == false`          | 警告：`"模型 'xxx' 已禁用"`                            |
| API Key 缺失 | 检查 APIKey 是否为空                   | 错误：`"模型 'xxx' 未配置 API Key"`                    |
| Agent 不存在 | 检查 `Agents().Get(agentName)`         | 错误：`"当前 Agent 'xxx' 不存在"`                      |
| 文件写入失败 | 检查文件系统权限                       | 错误：`"无法保存配置: {err}"`                          |
| 无参数执行   | `Args` 为空时                          | 提示：`"用法: /model <model_name>"` 并显示可用模型列表 |

#### ⚠️ 重要约束

1. **不保存到 mindx.json**
   - Model 是 Agent 的固有属性，不属于全局配置
   - mindx.json 只记录 `LastAgent`（用于启动时恢复上次对话的 Agent）
   - 通过 LastAgent → 加载 Agent 定义文件 → 自动获取其 Model

2. **不立即生效**
   - 当前正在运行的 Agent 实例不受影响
   - Agent 完成 TAO 循环后自动销毁
   - 下次创建新实例时从更新的定义文件加载

3. **原子性操作**
   - 如果写入失败，必须回滚内存中的配置
   - 确保定义文件不会处于不一致状态

### 6.4 /chat — 会话管理指令

#### 功能描述

用于管理对话会话，包括切换历史会话、创建新会话、清除当前会话等操作。

#### 语法格式

```
/chat <session_id | new | clear>
```

**参数**（三选一）：

| 参数类型     | 说明                         | 示例                |
| ------------ | ---------------------------- | ------------------- |
| `session_id` | 切换到指定的历史会话         | `/chat sess_abc123` |
| `new`        | 创建一个全新的空白会话       | `/chat new`         |
| `clear`      | 清除当前会话数据并创建新会话 | `/chat clear`       |

#### 触发条件与建议面板

当用户输入 `/chat` 时：
1. **立即激活 SessionSuggestion 面板**（见 §5.9.4）
2. **显示两类选项**：
   - **历史会话列表**：从 `SessionDB.ListSessions()` 获取（最多 10 条）
   - **特殊操作项**：固定显示 `[new]` 和 `[clear]`
3. **支持 ID 过滤**：输入 `/chat sess_ab` 只匹配包含该子串的会话

#### 行为流程

##### 场景 A：切换到历史会话

```
1. 用户输入: "/chat"
   ↓
2. 显示 SessionSuggestion 面板
   ┌─────────────────────────────┐
   │ > sess_abc123  ... (当前)   │
   │   sess_def456  ...         │
   │   [new]                     │
   │   [clear]                   │
   └─────────────────────────────┘
   ↓
3. 用户选择 "sess_def456" + Enter
   ↓
4. SlashCommandMsg { Name: "chat", Args: ["sess_def456"] }
   ↓
5. rootModel.handleSlashCommand()
   ↓
6. app.SwitchSession("sess_def456")
   ├── 从 SessionDB 加载会话元数据
   ├── 设置 app.CurrentSessionMeta(meta)
   ├── ConversationPanel 加载历史消息（如果有）
   └── StatusBar 显示: "Session: sess_def456"
   ↓
7. 保存到 ~/.mindx/mindx.json
   └── config.LastSessionID = "sess_def456"
   └── config.Save()
```

##### 场景 B：创建新会话

```
1. 用户选择 [new] 或输入 "/chat new" + Enter
   ↓
2. SlashCommandMsg { Name: "chat", Args: ["new"] }
   ↓
3. app.CreateSession(currentAgentName)
   ├── 生成新的 SessionID
   ├── 创建会话目录结构
   ├── 初始化 SessionMeta
   └── 返回 *core.SessionInfo
   ↓
4. app.SetCurrentSessionMeta(newSession)
   ↓
5. UI 更新
   ├── ConversationPanel 清空（准备接收新消息）
   └── StatusBar 显示: "Session: {new_session_id} (新建)"
   ↓
6. 保存到 mindx.json
   └── config.LastSessionID = newSessionID
   └── config.LastAgent = currentAgentName  // 记录关联
   └── config.Save()
```

##### 场景 C：清除当前会话（物理删除）

⚠️ **重要：`/chat clear` 执行的是不可逆的物理删除操作**

```
1. 用户选择 [clear] 或输入 "/chat clear" + Enter
   ↓
2. SlashCommandMsg { Name: "chat", Args: ["clear"] }
   ↓
3. app.ClearCurrentSession()
   ├── 获取当前 sessionID（如 "sess_abc123"）
   │
   ├── [步骤 1] 物理删除磁盘数据
   │   ├── SessionDB.Delete(sessionID)
   │   │   └── os.RemoveAll(sessionDir)  // 删除整个会话目录
   │   │       ├── session.yml (元数据)
   │   │       ├── messages/ (历史消息)
   │   │       └── workspace/ (工作区快照)
   │   └── 如果删除失败 → 返回错误，中止操作
   │
   ├── [步骤 2] 创建全新会话
   │   └── app.CreateSession(currentAgentName)
   │       ├── 生成新的 SessionID（如 "sess_xyz789"）
   │       ├── 创建新的会话目录结构
   │       └── 初始化空白的 SessionMeta
   │
   └── [步骤 3] 更新运行时状态和 UI
       ├── app.SetCurrentSessionMeta(newSession)
       ├── ConversationPanel.Clear()  // 清空所有显示内容
       └── StatusBar 显示: "Session: sess_xyz789"
   ↓
4. 保存到 mindx.json
   └── config.LastSessionID = "sess_xyz789"
   └── config.Save()
   ↓
5. 额外反馈
   └── NotificationBar 提示:
       "✅ 已清除旧会话 sess_abc123 (数据已永久删除)"
       "已创建新会话 sess_xyz789"
```

**⚠️ 数据安全警告：**

| 项目         | 说明                                           |
| ------------ | ---------------------------------------------- |
| **删除范围** | 整个会话目录（元数据 + 消息记录 + 工作区快照） |
| **可恢复性** | ❌ **不可恢复**（无回收站机制）                 |
| **确认机制** | 当前版本**不提供确认对话框**，直接执行         |
| **建议改进** | 未来版本应考虑添加 ChoicesPanel 确认步骤       |

**实现要求：**

1. **原子性操作**
   - 先执行删除，再创建新会话
   - 如果删除失败，**不得创建新会话**（避免状态不一致）
   - 如果创建失败，旧会话已被删除（用户需手动处理）

2. **错误处理优先级**
   ```
   P0: 获取当前 sessionID 失败 → 错误："无法获取当前会话"
   P1: 物理删除失败              → 错误："删除失败: {err}"（中止）
   P2: 创建新会话失败            → 错误："创建新会话失败: {err}"
       （此时旧会话已删除，需要手动恢复或重新启动）
   ```

3. **日志记录**
   ```go
   app.logger.Warn("session physically deleted",
       "session_id", oldSessionID,
       "reason", "user requested /chat clear",
   )
   ```

---

⚠️ **关键约束：`/chat new` 和 `/chat clear` 必须同步更新以下状态**

##### 必须更新的内存状态

```go
type App struct {
    currentSessionMeta *core.SessionInfo  // ✅ 必须更新
    master             *goreact.Agent      // ✅ 可能需要重新创建
}

type rootModel struct {
    conversation       *conv.ConversationPanel  // ✅ 必须清空/重新加载
    statusBar          *statusbar.StatusBar     // ✅ 必须更新 SessionID 显示
}
```

##### 必须持久化的配置字段

```go
type MindxConfig struct {
    LastAgent     string  // ✅ /chat new 时更新（如果 Agent 变更）
    LastSessionID string  // ✅ 所有 /chat 操作都必须更新
}
```

**保存时机**：在会话操作完成后**立即保存**，不要延迟或批量保存。

#### 错误处理

| 错误情况     | 处理方式                   | 用户反馈                                              |
| ------------ | -------------------------- | ----------------------------------------------------- |
| 会话不存在   | 检查 SessionDB 中是否存在  | 错误：`"会话 'xxx' 不存在"`                           |
| 会话加载失败 | 文件损坏或权限问题         | 错误：`"无法加载会话: {err}"`                         |
| 创建会话失败 | 目录创建失败或 ID 冲突     | 错误：`"创建会话失败: {err}"`                         |
| 无参数执行   | Args 为空                  | 提示：`"用法: /chat <id\|new\|clear>"` 并显示建议面板 |
| 参数无效     | 既不是 ID 也不是 new/clear | 错误：`"无效参数: '{arg}'"`                           |

### 6.5 指令注册与扩展机制

#### 注册接口

所有斜杠命令通过 `SlashCommandRegistry` 统一注册：

```go
// internal/client/command/registry.go
type SlashCommand struct {
    Name        string             // 命令名称（不含 /）
    Description string             // 简短描述（用于建议面板）
    Handler     func(args []string) CommandResult  // 执行函数
    HasSuggestion bool            // 是否有建议面板
}

type CommandResult struct {
    Message string                 // 结果消息（用于 NotificationBar）
    Success bool                  // 是否成功
}
```

#### 内置命令注册表

```go
func BuiltinCommands() *SlashCommandRegistry {
    registry := NewSlashCommandRegistry()
    
    registry.Register(SlashCommand{
        Name:        "help",
        Description: "显示帮助信息",
        Handler:     handleHelp,
        HasSuggestion: false,
    })
    
    registry.Register(SlashCommand{
        Name:        "model",
        Description: "切换当前模型",
        Handler:     handleModel,
        HasSuggestion: true,  // 启用 ModelSuggestion
    })
    
    registry.Register(SlashCommand{
        Name:        "chat",
        Description: "管理会话",
        Handler:     handleChat,
        HasSuggestion: true,  // 启用 SessionSuggestion
    })
    
    return registry
}
```

#### 扩展指南

添加新命令的步骤：

1. **定义命令处理器**
   ```go
   func handleMyCommand(args []string) CommandResult {
       // 实现逻辑
       return CommandResult{Message: "完成", Success: true}
   }
   ```

2. **注册到 Registry**
   ```go
   registry.Register(SlashCommand{
       Name:        "mycommand",
       Description: "我的自定义命令",
       Handler:     handleMyCommand,
       HasSuggestion: false,  // 或 true 如果需要建议面板
   })
   ```

3. **（可选）实现自定义 Suggestion**
   - 如果 `HasSuggestion: true`，需要在 InputArea 中添加对应的建议逻辑
   - 参考 ModelSuggestion 和 SessionSuggestion 的实现

### 6.7 配置文件设计原则 — mindx.json vs Agent 定义文件

⚠️ **重要：理解 `mindx.json` 和 Agent 定义文件的职责边界**

#### 数据归属分类

| 数据类型       | 存储位置                     | 文件格式 | 管理者                            |
| -------------- | ---------------------------- | -------- | --------------------------------- |
| **全局配置**   | `~/.mindx/mindx.json`        | JSON     | App 启动/关闭时自动读写           |
| **Agent 定义** | `~/.mindx/agents/{name}.yml` | YAML     | 用户手动编辑 或 `/model` 命令修改 |
| **会话数据**   | `~/.mindx/sessions/{id}/`    | 目录结构 | SessionDB 自动管理                |

#### mindx.json 的字段职责

```go
type MindxConfig struct {
    Version       int           `json:"version"`
    Initialized   bool          `json:"initialized"`

    // ✅ 核心字段：用于启动时恢复上下文
    LastAgent     string        `json:"last_agent,omitempty"`      // 上次对话的 Agent
    LastSessionID string        `json:"last_session_id,omitempty"` // 上次对话的 SessionID

    // ⚠️ 兜底字段：仅在首次启动且无历史记录时使用
    DefaultModel  string        `json:"default_model,omitempty"`   // 系统默认模型（安装时设置）

    // 🔧 技术配置
    EmbedderModel string        `json:"embedder_model,omitempty"`
    Daemon        DaemonConfig  `json:"daemon"`
    Python        PythonConfig  `json:"python"`
}
```

**字段加载优先级（启动时）：**

```
1. LastAgent (最高优先级)
   └── 如果存在 → 使用该 Agent 作为当前 Agent
   └── 如果不存在 ↓

2. DefaultModel (兜底值)
   └── 仅当 LastAgent 为空时使用
   └── 用于首次启动或完全重置后的场景
   └── 通常在安装向导 (/doctor) 中由用户选择
```

**❌ 为什么不保存 LastModel 到 mindx.json？**

| 方案                             | 问题                                        |
| -------------------------------- | ------------------------------------------- |
| ❌ 在 mindx.json 中保存 LastModel | 违反单一职责原则，Model 是 Agent 的固有属性 |
| ❌ 每次切换模型都更新 mindx.json  | 导致不必要的 IO 操作和配置膨胀              |
| ✅ 通过 Agent 定义文件管理 Model  | 符合「Agent 是一等公民」的设计理念          |

#### Agent 定义文件的结构 (~/.mindx/agents/architect.yml)

```yaml
description: "专注于系统设计和架构的 AI 助手"
model: claude-sonnet-4-20250514    # ← Model 是 Agent 的固定属性
```

| 操作方式           | 触发条件          | 变更持久性                  |

## 7. 组装层 — rootModel

rootModel 是唯一的 `tea.Model`，负责组合所有子组件。

```
┌──────────────────────────────────────────────────────────────┐
│                        rootModel                             │
├──────────────────────────────────────────────────────────────┤
│  职责                                                       │
│  ────                                                       │
│  • 持有所有子组件实例                                        │
│  • 在 Update 中按消息类型分发到对应子组件                    │
│  • 跨组件协调（如 WindowResize 需要通知所有组件）            │
│  • 初始化 App + 会话                                         │
│  • 启动 consumeEvents goroutine                              │
│                                                              │
│  组合关系                                                   │
│  ────────                                                   │
│  rootModel                                                    │
│    ├── conversation: *ConversationPanel                      │

│    ├── notifBar: *NotificationBar                            │
│    ├── choices: *ChoicesPanel                                │
│    │                                                         │
│    ├── app: *core.App                          ← 外部依赖   │
│    ├── chatManager: *chatSessionManager        ← 持久化     │
│    ├── registry: *SlashCommandRegistry         ← 命令注册   │
│    └── (goroutine 相关)                                       │
│        ├── outputCh: chan tea.Msg                            │
│        ├── currentCancel: context.CancelFunc                 │
│        └── executing: bool                                   │
│                                                              │
│  Update 消息分发逻辑（伪代码）                                │
│  ────────────────────────                                    │
│  Update(msg):                                                │
│    switch msg:                                               │
│      AgentEvent:        conversation.Update(msg)             │

│                          conversation.Update(msg)             │
│                          input.Update(msg)                    │
```

## 8. 数据流

### 8.1 主消息流 — 用户发送消息到 Agent 应答

```
User
  │
  ▼
InputArea.HandleKey(Enter)
  │
  ├── 产生 UserSend{Text}
  │
  ▼
rootModel.Update(UserSend)
  │
  ├── 1. conversation.CreateAnswer(sessionID, agentName)
  ├── 2. goroutine: agent.Ask(sessionID, text)
  ├── 3. goroutine: consumeEvents(eventCh, sessionID)
  │
  ▼
agent.Ask() → eventCh(ReactEvent)
  │
  ▼
consumeEvents goroutine
  │  for event := range eventCh:
  │    switch event.Type:
  │      ThinkingDelta → outputCh ← ThinkingDeltaMsg
  │      ActionStart   → outputCh ← ActionStartMsg
  │      ActionResult  → outputCh ← ActionResultMsg
  │      FinalAnswer   → outputCh ← FinalAnswerMsg
  │      Error         → outputCh ← AgentErrorMsg
  │  after loop → outputCh ← SessionDoneMsg
  │
  ▼
rootModel.Update(ThinkingDeltaMsg)
  │
  ├── conversation.Update(ThinkingDeltaMsg)
  │     └── 在当前 Answer 的 PendingThink 中追加 Content
  │
  ▼ (循环反复)
  │
  rootModel.Update(SessionDoneMsg)
  │
  ├── conversation.Update(SessionDoneMsg)
  │     └── 标记 Answer 为 Done
  ├── executing = false
```

### 8.2 闪烁动画流

```
ConversationPanel 在 ActionStart/ThinkingDelta 时
  返回 Cmd: Tick
      │
      ▼
  rootModel 收到 Tick 消息
      │
      ├── conversation.Update(Tick)
      │     └── 切换 BlinkOn 状态
      ├── 如果仍有需要动画的状态，返回下一个 Tick Cmd
      │
      ▼
  conversation.View() 根据 BlinkOn 交替闪烁图标

动画规则详见 §5.8：
  • 思考区：BlinkOn 交替 白色 ↔ 蓝色 (#E0E0E0 ↔ #4FC3F7)
  • 工具区：BlinkOn 交替 白色 ↔ 绿色 (#E0E0E0 ↔ #4CAF50)
  • 停止条件：ThinkingDone + 所有 ActionStep 非 Executing
```

### 8.3 窗口尺寸变化流

```
tea.WindowSizeMsg
  │
  ▼
rootModel.Update(WindowSizeMsg)
  │
  ├── statusBar.Update(WindowResize)   → 更新 Width + ConnState
  ├── conversation.Update(WindowResize)→ 更新 viewport 尺寸
  ├── input.Update(WindowResize)       → 更新 textarea 宽度
  └── notifBar.Update(WindowResize)    → 更新通知栏宽度
```

## 9. 与当前架构的关键差异

| 维度             | 当前架构 (V1)                              | 新架构 (V2)                                |
| ---------------- | ------------------------------------------ | ------------------------------------------ |
| **组件接口**     | 4+ 种不同签名                              | 统一 `Update(Msg) → (Self, Cmd)`           |
| **状态变更**     | setter + Update 双路径                     | 仅通过 Update                              |
| **事件路由**     | `routeToAnswer` switch string              | 强类型消息直接分发                         |
| **Answer**       | 既是数据又是组件                           | 纯数据 AnswerData + ConversationPanel 管理 |
| **goroutine→UI** | 调 answer.SetXxx() + trySend               | 仅 trySend(msg)                            |
| **消息类型**     | `agentAnswerUpdateMsg{contentType string}` | 细分到具体语义的消息                       |
| **新增事件**     | 改 switch case                             | 新建消息类型 + 新增 Update 分支            |
| **显示规范**     | 无独立文档，散布在 View 代码中             | §5 独立章节，所有组件约束统一              |
| **动画控制**     | time.Time Tick 直接转发                    | Tick 经 ConversationPanel.Update 统一管理  |
| **可测试性**     | 需要 mock setter                           | `Update(msg) → assert View()`              |
