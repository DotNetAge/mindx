## 6. 指令系统 — Slash Commands 规格说明

MindX CLI 提供了一套以 `/` 开头的斜杠命令，用于执行特定操作。
指令系统遵循「建议优先」的设计理念：所有指令都支持 Tab 补全和上下文感知的建议面板。

---

### 6.1 指令分类

| 分类       | 命令      | 说明            | 建议面板            |
| ---------- | --------- | --------------- | ------------------- |
| **配置类** | `/model`  | 切换 Agent 模型 | ✅ ModelSuggestion   |
| **会话类** | `/chat`   | 管理会话        | ✅ SessionSuggestion |
| **系统类** | `/doctor` | 调起安装向导    | ❌ 无需              |

> **注意：** `/help` 指令已移除，帮助功能将在后续版本以更完善的形式重新设计。

---

### 6.2 /doctor — 系统诊断与安装向导

#### 功能描述

调起 MindX 安装向导（`setup.RunFirstRunWizard()`），用于首次安装、配置修复、环境检测等场景。

#### 语法格式

```
/doctor
```

**参数**：无

**触发条件**：输入 `/doctor` 后按 Enter 执行

#### 行为流程

```
用户输入: "/doctor" + Enter
    ↓
SlashCommandMsg { Name: "doctor", Args: [] }
    ↓
rootModel.handleSlashCommand()
    ↓
调用 setup.RunFirstRunWizard(modelsPath, agentsDir, workspaceDir, mindxConfig)
    ↓
启动交互式安装向导（独立 bubbletea Program）
```

#### 安装向导步骤（基于 setup/wizard.go 实现）

向导包含 **6 个配置步骤**，按顺序执行：

| 步骤 | 名称 | 功能说明 | 跳过条件 |
|------|------|---------|---------|
| **Step 0** | 选择默认模型 | 从 models.yml 列表中选择 AI 模型 | 已有 DefaultModel 配置时按 **S** 跳过 |
| **Step 1** | API Key 配置 | 输入选中模型的 API 密钥（密码模式） | 已有 API Key 时按 **S** 跳过 |
| **Step 2** | Daemon 后台服务 | 选择是否注册为开机自启动服务 | 已安装时自动跳过 |
| **Step 3** | Python 环境检测 | 检测 Python 版本，选择是否创建虚拟环境 | 已有 .venv 时按 **S** 跳过 |
| **Step 4** | 记忆体配置 | 下载 Chinese-CLIP Embedder 模型（用于 RAG） | 已有模型文件时按 **S** 跳过 |
| **Step 5** | 系统 PATH 配置（仅 Windows） | 将 mindx 目录添加到 PATH | 已在 PATH 中时按 **S** 跳过 |

#### 交互操作指南

每个步骤支持的操作：

| 按键 | 功能 |
|------|------|
| **↑↓** | 在选项列表中移动（Step 0） |
| **←→** | 切换 Yes/No 选项（Step 2-5） |
| **Enter** | 确认当前选择，进入下一步 |
| **S** | 跳过此步骤（使用已有配置或默认值） |
| **Esc** | 退出向导（仅在未配置项时有效） |
| **Q / Ctrl+C** | 强制退出向导 |

#### 返回值处理

```go
// 向导返回的 firstRunResult 结构
type firstRunResult struct {
    SelectedModel string         // 用户选择的模型名称
    CredRef       string         // 模型的凭据引用名
    APIKey        string         // 用户输入的 API Key
    Err           error          // 错误信息（如有）

    DaemonSetup    bool           // 是否配置 Daemon 服务
    PythonSetup    bool           // 是否创建 Python 虚拟环境
    PythonInfo     core.PythonConfig // Python 环境信息
    EmbedderModel  string         // Embedder 模型文件名
    PathSetup      bool           // 是否添加到 PATH
}

// handleSlashCommand() 中的处理逻辑
result := setup.RunFirstRunWizard(...)
if result.Err != nil {
    NotificationBar.Show("❌ 向导失败: " + result.Err.Error())
} else {
    // 保存配置到 mindx.json
    mindxConfig.DefaultModel = result.SelectedModel
    credStore.Set(result.CredRef, result.APIKey)

    NotificationBar.Show("✅ 配置完成")
}
```

#### 适用场景

| 场景 | 触发方式 | 向导行为 |
|------|---------|---------|
| **首次安装** | 应用检测到 `mindx.json` 不存在或 `Initialized=false` | 完整执行所有 6 个步骤 |
| **配置修复** | 用户主动输入 `/doctor` | 从 Step 0 开始，已配置项可按 S 跳过 |
| **模型更换** | `/doctor` + 在 Step 0 选择新模型 | 仅重新配置 Model 和 API Key，后续步骤可跳过 |
| **环境变更** | Python 升级/重装后执行 `/doctor` | 重新检测 Python 环境（Step 3） |

#### 实现要求

1. **独立进程运行**
   - 向导通过 `tea.NewProgram(m, tea.WithoutSignals())` 启动独立的 bubbletea 程序
   - 不阻塞主 TUI 的 Update 循环

2. **状态保存**
   - 向导完成后立即将结果写入 `~/.mindx/mindx.json`
   - 更新字段：`DefaultModel`, `Initialized=true`, `Daemon`, `Python`, `EmbedderModel`

3. **中断恢复**
   - 用户按 Esc 或 Q 可随时退出
   - 退出后保留已完成的配置（不回滚）
   - 下次运行向导时可从断点继续（通过检测已有配置自动跳过已完成步骤）

### 6.3 /model — 模型切换指令

#### 功能描述

用于永久性更改当前 Agent 的模型配置。修改的是 **Agent 定义文件**（`~/.mindx/agents/{agent_name}.yml`），而非运行时状态。

⚠️ **核心设计原则：**
- Agent 在完成完整的 TAO（Think-Act-Observe）循环前不接受任何变更
- Agent 完成循环后会自动销毁，下次对话时从定义文件重新加载
- 因此 `/model` 只需修改磁盘上的定义文件，无需重建内存中的 Agent 实例

#### 语法格式

```
/model <model_name>
```

**参数**：
- `model_name` (必需): 目标模型的名称（如 `claude-sonnet-4-20250514`）

**示例**：
```bash
/model claude-opus-4-20250514    # 切换到 Claude Opus
/model gpt-4o                    # 切换到 GPT-4o
```

#### 触发条件与建议面板

当用户输入 `/model` 时：
1. **立即激活 ModelSuggestion 面板**（见 §5.9.3）
2. **列出所有可用模型**：从 `App.Models().List()` 获取
3. **支持缩写过滤**：输入 `/mod cl` 只显示包含 "cl" 的模型

#### 行为流程

##### 场景 A：通过建议面板选择（推荐）

```
1. 用户输入: "/model"
   ↓
2. 显示 ModelSuggestion 面板（列出所有可用模型）
   ↓
3. 用户按 ↑↓ 选择 "claude-sonnet-4-20250514"
   ↓
4. 用户按 Enter
   ↓
5. 发送 SlashCommandMsg { Name: "model", Args: ["claude-sonnet-4-20250514"] }
   ↓
6. rootModel.handleSlashCommand()
   ↓
7. 调用 app.SwitchAgentModel(currentAgentName, "claude-sonnet-4-20250514")
   ├── 从 Agents().Get(agentName) 获取 Agent 配置对象
   ├── 修改配置对象的 Model 字段为新值
   ├── 将更新后的配置写回定义文件 (~/.mindx/agents/{name}.yml)
   └── 清空 agentCache[agentName] 缓存（强制下次重新加载）
   ↓
8. StatusBar 显示: "✅ 已更新 {agent} 模型为: claude-sonnet-4-20250514"
   └── 提示: "下次对话时生效"
```

##### 场景 B：直接输入完整命令

```
1. 用户输入: "/model gpt-4o" + Enter
   ↓
2. 发送 SlashCommandMsg { Name: "model", Args: ["gpt-4o"] }
   ↓
3. 同场景 A 的步骤 6-8
```

#### 关键实现细节

**为什么不需要重建 Agent？**

```go
// Agent 生命周期说明：
//
// 1. 用户发送消息
//    ↓
// 2. app.ResolveAgent(name) 或 app.getMaster()
//    ├── 检查 agentCache 命中？
//    │   ├── 命中 → 返回缓存实例（复用）
//    │   └── 未命中 → 从定义文件加载 → 创建新实例 → 缓存
//    ↓
// 3. agent.Ask(sessionID, message)
//    ↓
// 4. Agent 执行 TAO 循环 (Think → Act → Observe)
//    ↓
// 5. 循环完成 → 返回 FinalAnswer
//    ↓
// 6. 下次调用时重复步骤 2（如果缓存被清除则重新加载）
//

// 因此 SwitchModel 只需：
func (app *App) SwitchAgentModel(agentName, modelName string) error {
    // 1. 获取 Agent 配置（从注册表，非内存实例）
    cfg := app.Agents().Get(agentName)
    if cfg == nil {
        return fmt.Errorf("agent %q 不存在", agentName)
    }

    // 2. 验证目标模型可用
    model := app.Models().Get(modelName)
    if model == nil || !model.Enabled {
        return fmt.Errorf("模型 %q 不可用", modelName)
    }

    // 3. 修改配置对象的 Model 字段
    oldModel := cfg.Model
    cfg.Model = modelName

    // 4. 写回定义文件（永久性变更）
    agentFile := filepath.Join(app.settings.AgentsDir(), agentName+".yml")
    if err := writeAgentConfig(agentFile, cfg); err != nil {
        // 回滚：恢复旧值
        cfg.Model = oldModel
        return fmt.Errorf("保存失败: %w", err)
    }

    // 5. 清除该 Agent 的缓存（强制下次重建）
    app.agentMu.Lock()
    delete(app.agentCache, agentName)
    app.agentMu.Unlock()

    // 6. 如果当前 master 就是被修改的 Agent，也清除 master 缓存
    if app.currentAgentName() == agentName {
        app.masterMu.Lock()
        app.master = nil
        app.masterMu.Unlock()
    }

    app.logger.Info("agent model updated",
        "agent", agentName,
        "old_model", oldModel,
        "new_model", modelName,
    )

    return nil
}
```

#### 错误处理

| 错误情况     | 处理方式                               | 用户反馈                                               |
| ------------ | -------------------------------------- | ------------------------------------------------------ |
| 模型不存在   | 检查 `Models().Get(name)` 是否返回 nil | NotificationBar 显示错误：`"模型 'xxx' 不存在"`        |
| 模型未启用   | 检查 `model.Enabled == false`          | 警告：`"模型 'xxx' 已禁用"`                            |
| API Key 缺失 | 检查 APIKey 是否为空                   | 错误：`"模型 'xxx' 未配置 API Key"`                    |
| Agent 不存在 | 检查 `Agents().Get(agentName)`         | 错误：`"当前 Agent 'xxx' 不存在"`                      |
| 文件写入失败 | 检查文件系统权限                       | 错误：`"无法保存配置: {err}"`                          |
| 无参数执行   | `Args` 为空时                          | 提示：`"用法: /model <model_name>"` 并显示可用模型列表 |

#### ⚠️ 重要约束

1. **不保存到 mindx.json**
   - Model 是 Agent 的固有属性，不属于全局配置
   - mindx.json 只记录 `LastAgent`（用于启动时恢复上次对话的 Agent）
   - 通过 LastAgent → 加载 Agent 定义文件 → 自动获取其 Model

2. **不立即生效**
   - 当前正在运行的 Agent 实例不受影响
   - Agent 完成 TAO 循环后自动销毁
   - 下次创建新实例时从更新的定义文件加载

3. **原子性操作**
   - 如果写入失败，必须回滚内存中的配置
   - 确保定义文件不会处于不一致状态

### 6.4 /chat — 会话管理指令

#### 功能描述

用于管理对话会话，包括切换历史会话、创建新会话、清除当前会话等操作。

#### 语法格式

```
/chat <session_id | new | clear>
```

**参数**（三选一）：

| 参数类型     | 说明                         | 示例                |
| ------------ | ---------------------------- | ------------------- |
| `session_id` | 切换到指定的历史会话         | `/chat sess_abc123` |
| `new`        | 创建一个全新的空白会话       | `/chat new`         |
| `clear`      | 清除当前会话数据并创建新会话 | `/chat clear`       |

#### 触发条件与建议面板

当用户输入 `/chat` 时：
1. **立即激活 SessionSuggestion 面板**（见 §5.9.4）
2. **显示两类选项**：
   - **历史会话列表**：从 `SessionDB.ListSessions()` 获取（最多 10 条）
   - **特殊操作项**：固定显示 `[new]` 和 `[clear]`
3. **支持 ID 过滤**：输入 `/chat sess_ab` 只匹配包含该子串的会话

#### 行为流程

##### 场景 A：切换到历史会话

```
1. 用户输入: "/chat"
   ↓
2. 显示 SessionSuggestion 面板
   ┌─────────────────────────────┐
   │ > sess_abc123  ... (当前)   │
   │   sess_def456  ...         │
   │   [new]                     │
   │   [clear]                   │
   └─────────────────────────────┘
   ↓
3. 用户选择 "sess_def456" + Enter
   ↓
4. SlashCommandMsg { Name: "chat", Args: ["sess_def456"] }
   ↓
5. rootModel.handleSlashCommand()
   ↓
6. app.SwitchSession("sess_def456")
   ├── 从 SessionDB 加载会话元数据
   ├── 设置 app.CurrentSessionMeta(meta)
   ├── ConversationPanel 加载历史消息（如果有）
   └── StatusBar 显示: "Session: sess_def456"
   ↓
7. 保存到 ~/.mindx/mindx.json
   └── config.LastSessionID = "sess_def456"
   └── config.Save()
```

##### 场景 B：创建新会话

```
1. 用户选择 [new] 或输入 "/chat new" + Enter
   ↓
2. SlashCommandMsg { Name: "chat", Args: ["new"] }
   ↓
3. app.CreateSession(currentAgentName)
   ├── 生成新的 SessionID
   ├── 创建会话目录结构
   ├── 初始化 SessionMeta
   └── 返回 *core.SessionInfo
   ↓
4. app.SetCurrentSessionMeta(newSession)
   ↓
5. UI 更新
   ├── ConversationPanel 清空（准备接收新消息）
   └── StatusBar 显示: "Session: {new_session_id} (新建)"
   ↓
6. 保存到 mindx.json
   └── config.LastSessionID = newSessionID
   └── config.LastAgent = currentAgentName  // 记录关联
   └── config.Save()
```

##### 场景 C：清除当前会话（物理删除）

⚠️ **重要：`/chat clear` 执行的是不可逆的物理删除操作**

```
1. 用户选择 [clear] 或输入 "/chat clear" + Enter
   ↓
2. SlashCommandMsg { Name: "chat", Args: ["clear"] }
   ↓
3. app.ClearCurrentSession()
   ├── 获取当前 sessionID（如 "sess_abc123"）
   │
   ├── [步骤 1] 物理删除磁盘数据
   │   ├── SessionDB.Delete(sessionID)
   │   │   └── os.RemoveAll(sessionDir)  // 删除整个会话目录
   │   │       ├── session.yml (元数据)
   │   │       ├── messages/ (历史消息)
   │   │       └── workspace/ (工作区快照)
   │   └── 如果删除失败 → 返回错误，中止操作
   │
   ├── [步骤 2] 创建全新会话
   │   └── app.CreateSession(currentAgentName)
   │       ├── 生成新的 SessionID（如 "sess_xyz789"）
   │       ├── 创建新的会话目录结构
   │       └── 初始化空白的 SessionMeta
   │
   └── [步骤 3] 更新运行时状态和 UI
       ├── app.SetCurrentSessionMeta(newSession)
       ├── ConversationPanel.Clear()  // 清空所有显示内容
       └── StatusBar 显示: "Session: sess_xyz789"
   ↓
4. 保存到 mindx.json
   └── config.LastSessionID = "sess_xyz789"
   └── config.Save()
   ↓
5. 额外反馈
   └── NotificationBar 提示:
       "✅ 已清除旧会话 sess_abc123 (数据已永久删除)"
       "已创建新会话 sess_xyz789"
```

**⚠️ 数据安全警告：**

| 项目         | 说明                                           |
| ------------ | ---------------------------------------------- |
| **删除范围** | 整个会话目录（元数据 + 消息记录 + 工作区快照） |
| **可恢复性** | ❌ **不可恢复**（无回收站机制）                 |
| **确认机制** | 当前版本**不提供确认对话框**，直接执行         |
| **建议改进** | 未来版本应考虑添加 ChoicesPanel 确认步骤       |

**实现要求：**

1. **原子性操作**
   - 先执行删除，再创建新会话
   - 如果删除失败，**不得创建新会话**（避免状态不一致）
   - 如果创建失败，旧会话已被删除（用户需手动处理）

2. **错误处理优先级**
   ```
   P0: 获取当前 sessionID 失败 → 错误："无法获取当前会话"
   P1: 物理删除失败              → 错误："删除失败: {err}"（中止）
   P2: 创建新会话失败            → 错误："创建新会话失败: {err}"
       （此时旧会话已删除，需要手动恢复或重新启动）
   ```

3. **日志记录**
   ```go
   app.logger.Warn("session physically deleted",
       "session_id", oldSessionID,
       "reason", "user requested /chat clear",
   )
   ```

---

⚠️ **关键约束：`/chat new` 和 `/chat clear` 必须同步更新以下状态**

##### 必须更新的内存状态

```go
type App struct {
    currentSessionMeta *core.SessionInfo  // ✅ 必须更新
    master             *goreact.Agent      // ✅ 可能需要重新创建
}

type rootModel struct {
    conversation       *conv.ConversationPanel  // ✅ 必须清空/重新加载
    statusBar          *statusbar.StatusBar     // ✅ 必须更新 SessionID 显示
}
```

##### 必须持久化的配置字段

```go
type MindxConfig struct {
    LastAgent     string  // ✅ /chat new 时更新（如果 Agent 变更）
    LastSessionID string  // ✅ 所有 /chat 操作都必须更新
}
```

**保存时机**：在会话操作完成后**立即保存**，不要延迟或批量保存。

#### 错误处理

| 错误情况     | 处理方式                   | 用户反馈                                              |
| ------------ | -------------------------- | ----------------------------------------------------- |
| 会话不存在   | 检查 SessionDB 中是否存在  | 错误：`"会话 'xxx' 不存在"`                           |
| 会话加载失败 | 文件损坏或权限问题         | 错误：`"无法加载会话: {err}"`                         |
| 创建会话失败 | 目录创建失败或 ID 冲突     | 错误：`"创建会话失败: {err}"`                         |
| 无参数执行   | Args 为空                  | 提示：`"用法: /chat <id\|new\|clear>"` 并显示建议面板 |
| 参数无效     | 既不是 ID 也不是 new/clear | 错误：`"无效参数: '{arg}'"`                           |

### 6.5 指令注册与扩展机制

#### 注册接口

所有斜杠命令通过 `SlashCommandRegistry` 统一注册：

```go
// internal/client/command/registry.go
type SlashCommand struct {
    Name        string             // 命令名称（不含 /）
    Description string             // 简短描述（用于建议面板）
    Handler     func(args []string) CommandResult  // 执行函数
    HasSuggestion bool            // 是否有建议面板
}

type CommandResult struct {
    Message string                 // 结果消息（用于 NotificationBar）
    Success bool                  // 是否成功
}
```

#### 内置命令注册表

```go
func BuiltinCommands() *SlashCommandRegistry {
    registry := NewSlashCommandRegistry()
    
    registry.Register(SlashCommand{
        Name:        "help",
        Description: "显示帮助信息",
        Handler:     handleHelp,
        HasSuggestion: false,
    })
    
    registry.Register(SlashCommand{
        Name:        "model",
        Description: "切换当前模型",
        Handler:     handleModel,
        HasSuggestion: true,  // 启用 ModelSuggestion
    })
    
    registry.Register(SlashCommand{
        Name:        "chat",
        Description: "管理会话",
        Handler:     handleChat,
        HasSuggestion: true,  // 启用 SessionSuggestion
    })
    
    return registry
}
```

#### 扩展指南

添加新命令的步骤：

1. **定义命令处理器**
   ```go
   func handleMyCommand(args []string) CommandResult {
       // 实现逻辑
       return CommandResult{Message: "完成", Success: true}
   }
   ```

2. **注册到 Registry**
   ```go
   registry.Register(SlashCommand{
       Name:        "mycommand",
       Description: "我的自定义命令",
       Handler:     handleMyCommand,
       HasSuggestion: false,  // 或 true 如果需要建议面板
   })
   ```

3. **（可选）实现自定义 Suggestion**
   - 如果 `HasSuggestion: true`，需要在 InputArea 中添加对应的建议逻辑
   - 参考 ModelSuggestion 和 SessionSuggestion 的实现

### 6.7 配置文件设计原则 — mindx.json vs Agent 定义文件

⚠️ **重要：理解 `mindx.json` 和 Agent 定义文件的职责边界**

#### 数据归属分类

| 数据类型       | 存储位置                     | 文件格式 | 管理者                            |
| -------------- | ---------------------------- | -------- | --------------------------------- |
| **全局配置**   | `~/.mindx/mindx.json`        | JSON     | App 启动/关闭时自动读写           |
| **Agent 定义** | `~/.mindx/agents/{name}.yml` | YAML     | 用户手动编辑 或 `/model` 命令修改 |
| **会话数据**   | `~/.mindx/sessions/{id}/`    | 目录结构 | SessionDB 自动管理                |

#### mindx.json 的字段职责

```go
type MindxConfig struct {
    Version       int           `json:"version"`
    Initialized   bool          `json:"initialized"`

    // ✅ 核心字段：用于启动时恢复上下文
    LastAgent     string        `json:"last_agent,omitempty"`      // 上次对话的 Agent
    LastSessionID string        `json:"last_session_id,omitempty"` // 上次对话的 SessionID

    // ⚠️ 兜底字段：仅在首次启动且无历史记录时使用
    DefaultModel  string        `json:"default_model,omitempty"`   // 系统默认模型（安装时设置）

    // 🔧 技术配置
    EmbedderModel string        `json:"embedder_model,omitempty"`
    Daemon        DaemonConfig  `json:"daemon"`
    Python        PythonConfig  `json:"python"`
}
```

**字段加载优先级（启动时）：**

```
1. LastAgent (最高优先级)
   └── 如果存在 → 使用该 Agent 作为当前 Agent
   └── 如果不存在 ↓

2. DefaultModel (兜底值)
   └── 仅当 LastAgent 为空时使用
   └── 用于首次启动或完全重置后的场景
   └── 通常在安装向导 (/doctor) 中由用户选择
```

**❌ 为什么不保存 LastModel 到 mindx.json？**

| 方案                             | 问题                                        |
| -------------------------------- | ------------------------------------------- |
| ❌ 在 mindx.json 中保存 LastModel | 违反单一职责原则，Model 是 Agent 的固有属性 |
| ❌ 每次切换模型都更新 mindx.json  | 导致不必要的 IO 操作和配置膨胀              |
| ✅ 通过 Agent 定义文件管理 Model  | 符合「Agent 是一等公民」的设计理念          |

#### Agent 定义文件的结构 (~/.mindx/agents/architect.yml)

```yaml
name: architect
role: "系统架构师"
description: "专注于系统设计和架构的 AI 助手"
model: claude-sonnet-4-20250514    # ← Model 是 Agent 的固定属性
system_prompt: |
  你是一位经验丰富的系统架构师...
skills:
  - design-patterns
  - api-design
  - database-modeling
```

**Model 字段的变更方式：**

| 操作方式           | 触发条件          | 变更持久性                  |
| ------------------ | ----------------- | --------------------------- |
| 手动编辑 yml 文件  | 用户直接编辑      | ✅ 永久生效                  |
| `/model` 命令      | 用户通过 TUI 执行 | ✅ 永久生效（修改 yml 文件） |
| 安装向导 `/doctor` | 首次安装或重置    | ✅ 永久生效                  |

**数据流图示：**

```
┌─────────────────────────────────────────────────────┐
│            配置数据流 — 启动与运行时                  │
```
