## 1. 从问题出发

一切设计始于问题。BionicBrain 不是凭空构想的理想架构，而是从当前 MindX 大脑的六个具体痛点中推导出来的。

### 1.1 意图识别是一锤子买卖

当前大脑的左脑做意图识别是"猜一次"——用户说一句话，左脑猜一个意图，猜错了就错了。现实中人脑处理模糊信息的方式是追问："你说的是 A 还是 B？"

这引出了第一个需求：**大脑需要一个澄清机制**， 说白了就需要是用LLM的多轮对话澄清用户意图，这样就不是一次性识别，而准确性就靠猜。

### 1.2 意图结构体不断膨胀

当前意图识别的输出是一个固定结构体，包含 10 个字段（answer、intent、keywords、send_to、has_schedule、schedule_name...）。每新增一种意图类型就加字段，结构体不断膨胀，左脑的 prompt 也跟着膨胀。换句话说 `Intent` 就是个“上帝对象”。

> 注：意图识别其实就是采用LLM对语意的理解，从自然语言中提取出我们需要的一个JSON结构的意图对象

这样做的原因：用户的提问第一时间是由本地的量化模型进行处理分析得出意图(`Intent`)（具体做法可以参考： ）, 
1. 实际运行发现量化模型中是无法支持多层嵌套的JSON输出。
2. 如果要量化模型从一句话中识别出人类的多种意图，如：显式意图，隐晦不明的含义，模型自己是在不编造答案的情况下是否有能力回答，用户是否要转发信息的要求，用户是否希望要做一个定期的计划等，一但多个意图同时要求识别，那量化模型就容易产生注意力分散的情况，输出的结果抖动得非常厉害，大大降低用户体验。

由此出发推论，我构思是否可以将 `Intent`对象看成一个上下文，如果一次调用LLM不能识别，那么可能就需要通过多次调用。例如，第一次是本地量化模型，如果识别出来的意图置信度低，那么可能就需要将LLM升级成云端大模型重新识别一次。从一开始就能完全了解用户真实想法。

> 采用本地模型的理由是希望将简单的，单一的任务由本地处理，只有复杂的逻辑推理才交由云端模型处理。

另外，对于连云端大模型的无没弄清的意图可能就需要通过多轮对话，让用户进一步澄清它的需求， 通过澄清Intent的置信度就可以提高，相关的字段也得以补充。


### 1.3 左右脑严格串行

当前架构中，左脑想完右脑才开始。右脑做深层思考时（可能几十秒），左脑完全闲置。而左脑运行在本地，右脑运行在云端，它们本可以并行工作。

其实这显得很多余的，本地量化模型的调用是串行的，队列化的，分左右脑也无法达到并行的目的。
另一个问题就是当前调用管线非常复杂，导致越来越难以扩展。

问题：怎样处理会更具扩展性，更加合理？

### 1.4 错误处理全有或全无

当前代码中，一个环节失败整条链路断裂——`if err != nil return`。但人脑不是这样的：你不会因为想不起一个词就整句话说不出来。

这引出了第四个需求：**失败应该触发降级，而不是中断**。

### 1.5 记忆、工具、用量统计是基础设施

当前架构中，Memory、Skills/MCP、TokenUsage 都是作为外部依赖注入大脑的。大脑"使用"它们，但它们不"属于"大脑。记忆是大脑的海马体，不是旁边的硬盘；行动能力是大脑的运动皮层，不是外接的机械臂。

这引出了第五个需求：**将大脑的能力内聚起来**。

### 1.6 情感盲区

当前大脑只识别"用户要做什么"，完全忽略"用户怎么说的"。但在中文世界，同样的文字加一个标点、换一个上下文，意思可能天差地别：

- "好的" → 可以是同意、敷衍、不耐烦
- "你真行" → 可以是夸奖、也可以是讽刺
- "呵呵" → 在不同语境下意思完全不同
- "帮我查一下天气" vs "帮我查一下天气！！" → 平静的请求 vs 焦急的需求

中文的语境依赖性极强，语气、标点、用词习惯、上下文情绪都是意图的组成部分。一个不能感知情感的大脑，意图识别就是在瞎猜——它可能猜对了"做什么"，但猜错了"怎么做"。焦急的用户需要简洁直接的回答，而不是慢悠悠的详细解释。

这引出了第六个需求：**大脑需要情感感知能力，它不是锦上添花，而是准确理解意图的前提**。说起来复杂，其实也应该是一次对话就可以获取到的属性：“用户的情感”

这个情感因数除了可以作为澄清用户意图的另一个修正手段，也应该存入到记忆中，同样可以使得记忆的向量搜索更加精准；


### 1.7 Skill 概念理解错误

当前 MindX 对 Skill 的概念理解与标准Skill [agentskills.io](https://agentskills.io/specification)的定义 存在严重偏差：

- **标准定义**：Skill 是通用能力描述，定义"做什么"和"何时使用"，是一种声明式规范，是一套资源包，一套SOP
- **当前实现**：Skill 被等同于可执行工具，包含具体执行逻辑和依赖管理，是一种命令式实现，原有的SKILL就应该坍缩为 `Tools`，而且是内部Tools, 用户不需要扩展这个Tools，因为这个Tools只是一个Go方法，如果向用户开放只会增加用户学习成本。如果需要扩展 MindX的 单项工具执行能力可以通过MCP达到。

因此：
1. 重新设计SKILL系统，使之符合 https://agentskills.io/specification 的定义，它应该是运行时由LLM去读取SKILL.md文件了解整个操作过程，然后从 Tools 库或 MCP 库中查找工具，接下来的一轮或者多轮的对话准备对应的工具集；
2. SKILL的使用是动态而且智能的。但要解决LLM不可能每次对话都去将系统中所有的SKILLS的 SKILL.md 读一次，如果系统中有500个 SKILL 这种Tokens消耗量是不可接受的，而且运行速度也是极为低下的，关于这一点可以去网上查一下LangChan或其它开源框架是怎么实现的。
3. 要将当下的SKILLS重新按照 https://agentskills.io/specification 的要求规范化，考虑如何对其向量化，或者什么高速索引的操作，以避免让LLM阅读大量的SKILL.md
4. 可以保留当前实现的MindX的MCP集成方法，使用起来要让调用法感觉与调用 Tools 是一样的，但 Tools和 MCP是表现形式上相似但本质上是完全不同的两种工具（在LLM的调用中它们又会变成同一样东西 如：OpenAI的`Tools`参数）
