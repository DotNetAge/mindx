# 目录的定义

- 用户目录：`~/.mindx` ，用于存储用户的配置文件和数据；
  - 配置目录: `~/.mindx/settings`
    - 模型配置文件 `~/.mindx/settings/models.json`
  - 核心配置文件 `~/.mindx/mindx.json`
  - 会话目录: `~/.mindx/sessions` 用于对会话进行隔离，每个会话都有自己的目录，用于存储会话相关的文件；
    - 会话目录的路径是：`<用户目录>/sessions/<会话ID>`
  - 技能目录: `~/.mindx/skills` 用于存储用户的技能文件；
  - Agent目录: `~/.mindx/agents` 用于存储用户的Agent文件；
- 工作目录：是 `mindx` 的当前执行位置，也就是 `os.Getwd()` 返回的目录；

以上就是当前MindX与GoReact的共同目录结构的定义；

---
当前的目录定义方式带来的问题：

- [ ] 沙箱隔离：到底是应该根据当前工作目录来隔离还是根据当前会话进行隔离，会话的最初设计只是为了能记录完整的不会丢失的会话数据，以及提供给大模型与用户之对话过程中产生的一些临时文件提供存储空间；
- [ ] `run_script` 的执行目录，当前应该是有歧义的，`run_script` 的执行当前是定义不清的，一个是Script的原初文件位置，第二，如果 script 是python那么就还应该指定当前脚本执行的工作目录；但脚本的工作目录与整个项目的工作目录本来应该就是两个不同的目录，国为 scripts的工作目录必然是 Skills/<skill_name>/scripts/<script_name>下的一个文件，而的MindX的工作目录则是执行`mindx`时获取到用户当前的执行目录；这里一定要搞清，因为这个问题又会与沙箱问题相关联；
- [ ] Demon 需要Hold住与Client端完全一至的Workspace, 否则对于计划任务的执行就会产生目录的偏移，可能会导致文件找不到或目录不正确的错误；因此，客户端是通过`os.Getwd()`来获取当前工作目录，而需要有一个手段来设置Demon的工作目录，以确保Demon与Client端是完全保持一至。
  - 思路1: 将 SessionID 与 工作目录绑定，一个会话就必须与一个工作目录绑定；
