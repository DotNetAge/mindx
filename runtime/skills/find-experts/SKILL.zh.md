# 寻找专家（find-experts）— 技能中文对照指南

> 本文件是 `find-experts` SKILL.md 的中文翻译与注释版，仅供开发者阅读对照。
> LLM 实际执行时使用的是同目录下的 SKILL.md（英文版）。

---

## 触发决策

当满足以下**任一条件**时使用本技能：

- 任务需要的专业知识超出你的核心能力范围
- 任务可分解为多个独立领域，适合并行委派处理
- 用户明确请求引入专业角色（如"找个安全专家审查一下"）

以下情况**不要**使用本技能：

- 你可以用自己的工具和技能直接完成任务
- 任务是简单的文件编辑、搜索或单条命令执行
- 委派的开销比自己直接做还大

## 模式选择器

根据任务复杂度选择执行模式：

```
任务复杂度
    │
    ├─ 单一领域，一次性咨询
    │   → 模式 0：单专家
    │
    ├─ 多个独立领域，无依赖关系
    │   → 模式 1：多专家并行
    │
    └─ 复杂的多阶段工作，存在相互依赖
        → 模式 2：团队协作编排
```

---

## 模式 0：单专家

**适用场景**：只需要一个专家的意见或操作。

**流程**：

1. `mindx agent list --json` — 选择一个匹配领域的专家
2. `sub-agent(agent_name, task)` — 委派（task 必须自包含）
3. `collect-results(task_ids)` — 阻塞等待完成
4. 检查结果 → 交付或自行补充

**协调规则**：

- 将 `task` 参数写成自包含的任务简报。子代理看不到我们的对话上下文。
- 简报中需包含：用户的原始请求、具体的交付格式、约束条件。
- 如果结果不充分，要么用更精确的描述重试，要么自己补充缺口。

---

## 模式 1：多专家并行

**适用场景**：多个独立领域需要同时推进工作（例如安全审计 + 性能审查 + 测试覆盖率分析）。

**流程**：

1. `mindx agent list --json` — 识别所有需要的专家（每个领域一个）
2. 在 **同一个响应中** 发起 **所有** `sub-agent` 调用 — 它们会自动并行执行
3. `collect-results(task_ids)` — 一次性收集全部结果
4. `task-create` + `task-update(completed)` — 记录做了什么（可选但推荐）
5. 将结果综合成统一回答

**协调规则**：

- 同一响应中的多个 `sub-agent` 调用会自动并行。**不要**把它们拆分到多个响应中——那会导致串行化。
- 每个 `task` 简报必须是独立的、自包含的。简报之间不要交叉引用（如"看看另一个专家发现了什么"）。
- 如果某个子代理失败，需要决策：用更清晰的指令重试、自己替代完成，或者报告部分结果并说明原因。
- 收集结果时，不要盲目转发原始输出。向用户报告前必须验证完整性。

---

## 模式 2：团队协作编排

**适用场景**：复杂任务需要分阶段推进、角色分配、依赖跟踪和进度监控。

### 第 1 阶段：组建

```
team-create(
  team_name="<kebab-case 格式的名称>",
  description="<团队在做什么>",
  leader="<你的名字或指定的协调者>",
  members=["<专家-1>", "<专家-2>", ...],
  tasks=["<给成员-1 的任务>", "<给成员-2 的任务>", ...]  // 可选，自动轮询分配
)
```

**关键细节**：`tasks` 参数会自动创建 Task 记录并按轮询方式将 owner 分配给各成员。**优先使用此方式**，而非手动逐个 `task-create`。

如果需要自定义分配（非轮询），则在 `team-create` 中省略 `tasks`，然后手动：

```
task-create(subject, description)          // 创建时不指定 owner
task-update(task_id, owner="<专家名称>")   // 手动指定负责人
```

### 第 2 阶段：建立依赖（如需要）

如果某些工作必须在其他工作之前完成：

```
// "架构设计" 必须在 "后端实现" 之前完成
task-update(task_id=后端任务, addBlockedBy=[架构设计任务])

// "前端" 和 "测试" 都依赖 "API 设计"
task-update(task_id=前端任务, addBlockedBy=[API设计任务])
task-update(task_id=测试任务, addBlockedBy=[API设计任务])
```

系统会自动拒绝循环依赖。不需要手动检查。

### 第 3 阶段：按波次执行

按依赖顺序执行任务。同一波次中的独立任务一起派发：

```
// 波次 1：无前置依赖的任务
task-update(task_id=波次1任务列表, status="in_progress")
sub-agent(agent_name=..., task=...)  // 所有波次 1 的调用写在同一响应中
collect-results(task_ids=[...])
task-update(task_id=波次1任务列表, status="completed")

// 波次 2：依赖波次 1 的任务
task-update(task_id=波次2任务列表, status="in_progress")
sub-agent(...)  // 所有波次 2 的调用写在同一响应中
collect-results(...)
task-update(task_id=波次2任务列表, status="completed")

// 重复直到所有波次完成
```

### 第 4 阶段：验证和收尾

```
team-get-tasks(team_name="...")  // 确认全部已完成
team-delete(team_name="...")     // 清理
```

**协调规则**：

- 派发前始终标记 `in_progress`，收集结果后立即标记 `completed`。
- 每 3 个完成的任务会触发验证提醒——要采取行动（运行测试、审查文件）。
- 如果子代理结果不合格，用 `task-update(status="cancelled")` 并附上说明，然后重新委派或自己处理。
- 不要让任务卡在 `in_progress` 状态。如果被阻塞，调查原因或取消。

## 跨模式通用规则（始终适用）

| 规则 | 原因 |
|------|------|
| SubAgent 的 `task` 必须自包含 | 子代理完全看不到我们的对话上下文 |
| 多个 SubAgent 调用 = 并行 | 同一响应 → 并发执行；拆分响应 → 串行 |
| 每次 SubAgent 批次后都要 CollectResults | 结果存放在 ResultStore 中，不收集就是浪费 |
| 状态变化后立即 TaskUpdate | 不要攒批量更新；实时跟踪才能支持恢复 |
| 绝不盲目信任子代理输出 | 向用户转发前必须检查 |
| 如实给专家打分 | 虚高的评分会破坏未来的专家选择统计 |

## 质量门禁

在向用户交付最终结果之前，逐一检查：

1. **完整性**：结果是否覆盖了用户原始请求的所有部分？
2. **正确性**：是否有事实错误、幻觉或逻辑漏洞？
3. **可操作性**：用户能否直接使用这个结果？是否需要综合/格式化？
4. **归因**：清晰标注哪位专家贡献了哪个部分。

## 评分

每次使用专家后进行评分：

```bash
mindx agent score --agent-name "<名称>" --task "<描述>" --score N --notes "<评价>"
```

| 分数 | 含义 |
|------|------|
| 9-10 | 卓越 — 超出预期 |
| 7-8 | 良好 — 扎实的结果，可直接使用 |
| 5-6 | 及格 — 可用但有需要你填补的缺口 |
| 3-4 | 明显不足 — 需要大量补充工作 |
| 1-2 | 不可用 — 基本等于你重新做了一遍 |

务必诚实。虚高的评分会损害未来的专家筛选质量。

## 汇报格式

### 完成的情况

```
完成了。结果如下：

[结果内容]

贡献者：
- <专家名称> (<角色>)：<他/她交付的内容>
- <专家名称> (<角色>)：<他/她交付的内容>
- 自己：<你补充或综合的部分>
```

### 部分 / 有问题的情况

```
部分完成。以下是我能获取到的结果：

[已完成的部分]

缺口 / 问题：
- <对缺失或错误内容的描述>

建议的下一步：
- 方案 A：用 <调整后的方法> 重试
- 方案 B：换一个专家：<名称>（<理由>）
- 方案 C：我可以直接处理剩余部分
```

## 反模式

- 不要对自己专业范围内或你能更快完成的小事使用 `sub-agent`
- 在运行 `mindx agent list --json` 之前不要创建 Agent——可能已经有合适的专家
- 不要接受未经验证的子代理输出——向用户报告前必须检查
- 不要把并行的 `sub-agent` 调用拆分到多个响应中——那样会失去并行性
- 不要让任务无限期停留在 `in_progress` —— 完成或取消它们
- 不要跳过 `collect-results` —— 未收集的结果白白消耗资源
- 不要打虚高的分数——诚实的反馈能让专家池持续改进

## 与旧版本的对比

| 维度 | 旧版本 | 新版本 |
|------|--------|--------|
| 定位 | 工具教程 + 流程混合 | 纯粹的编排策略手册 |
| 信息来源 | 大量重复工具 Prompt | 假设 LLM 已知工具用法 |
| 核心增量 | 低（~30% 是信号） | 高（~90% 是信号） |
| 模式划分 | 无（只有线性步骤） | 三档模式（0/1/2）+ 决策树 |
| 团队流程 | team-create 后手动 task-create | 优先利用内置 tasks 自动分配 |
| 依赖管理 | 未提及 | addBlocks/addBlockedBy + 波次执行 |
| allowed-tools | 缺少 task-update 等 | 补全为完整工具集 |
| 错误恢复 | 未提及 | nudge 提醒 + cancel 降级策略 |
| 汇报格式 | 一句话带过 | 完整模板（成功 / 部分） |
| 反模式 | 4 条 | 7 条（覆盖并行性、资源清理等） |
