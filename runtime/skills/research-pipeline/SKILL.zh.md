# 研究管线 — 深度研究与知识生产技能

## 触发条件

**使用本技能的场景：** 用户需要深入、结构化的研究，包括但不限于：

- **市场分析** — 市场规模、TAM/SAM/SOM、增长趋势、细分市场动态
- **竞品情报** — 竞争对手图谱、功能对比、定位分析、赢/输模式
- **学术文献综述** — 系统性论文调研、引用网络分析、研究空白识别
- **投资尽职调查** — 目标公司深度研究、财务+技术+团队评估
- **技术评估** — 技术栈评估、新兴技术雷达、自研 vs 采购分析
- **行业报告** — 行业概览、监管环境、生态图谱、未来展望

**不要使用本技能的场景：** 快速查证、单问题查询、简单的信息检索。这些情况直接使用查询或简单的网络搜索即可。本技能专为产生结构化、可引用、可作决策依据的研究成果而设计。

---

## 核心原则：先问对问题，再谈方法

> 研究中最昂贵的错误，就是完美地回答了错误的问题。

本技能围绕一个核心理念构建：**问题的定义决定了答案的价值**。工具（网络搜索、知识图谱搜索、文档获取）是 LLM 与生俱来的能力——不需要教。需要教的是如何**问对问题、规划合适的深度、验证输出是否真正回答了所问的内容。**

本技能分为三个层次：

| 层次 | 内容 | 负责方 |
|------|------|--------|
| **方法论** | 如何定义问题、选择深度、构建分析 | 本技能（提示词） |
| **工具** | WebSearch、QuickSearch、QuickExplore、FindRelation、WebFetch | LLM 天生能力（不在此教授） |
| **编排** | 大型研究的多智能体协调 | 可选，由深度级别触发 |

---

## 第一阶段：问题定义

**目标：** 将模糊的需求转化为精确、可验证的研究目标。这是最重要的阶段——在此投入总时间的 10-15%。

### 步骤 1.1 — 四问提取框架

不要以"你想研究什么？"开头。用以下结构化序列来挖掘用户的真实需求：

#### Q1 — 决策锚定

问用户：

> "这项研究会用来做什么决策？你会根据结果采取什么行动？"

这是最重要的一个问题。它决定了：
- **精度要求** — 1000 万美元的投资决策 vs 单纯的好奇心，需要的严谨程度天差地别
- **输出形式** — 董事会演示、内部备忘录、个人知识积累
- **置信度门槛** — 你需要多确定？

锚定答案的例子：
- "我需要决定是否投资 X 公司" → 全面研究，高置信度要求
- "我在探索是否要进入亚太市场" → 深度研究，方向性置信度即可
- "我想了解向量数据库是怎么工作的" → 快速扫描，低风险

#### Q2 — 已知与未知分离

问用户：

> "关于这个话题，你已经知道什么？你具体想解决哪些未知问题？"

这可以避免在已知信息上浪费精力，将研究聚焦在真正的空白上。

将输出记录为：

```
已知：
- [事实 1]
- [事实 2]

未知（研究目标）：
- [空白 1]
- [空白 2]
```

#### Q3 — 一阶原理拆解

问用户：

> "抛开所有假设，这本质上是什么问题？"

引导用户将问题拆解到不可再分的基本元素。例如，"医疗领域的 AI 智能体"可以拆解为：
- 谁是买单方？（医院系统？保险公司？患者？）
- 解决了什么具体问题？（降本？提效？可及性？）
- 为什么是现在？（监管变化？技术成熟？市场压力？）
- 这件事要成立，需要什么条件为真？

这一步可以防止用户自身的预设框架带偏研究的方向。

#### Q4 — 深度校准

问用户：

> "如果结论是错的，代价是什么？如果这项研究让你做出了错误判断，会发生什么？"

根据答案确定研究级别：

| 错误结论的后果 | 研究级别 | 方法 |
|---------------|----------|------|
| 轻微不便 | **Level 1 — 快速扫描** | 单智能体，<10 个来源，半天 |
| 浪费资源/精力 | **Level 2 — 深度研究** | 单智能体，10-30 个来源，1-2 天 |
| 重大财务/战略损失 | **Level 3 — 全面研究** | 多智能体，30+ 来源，交叉验证 |
| 持续的竞争劣势 | **Level 4 — 持续情报** | 多智能体 + 定期刷新 |

### 步骤 1.2 — 记录研究协议

四问之后，综合成一份简洁的协议：

```markdown
## 研究协议

**决策目标：** [这项研究将服务于什么决策？]
**核心问题：** [单一、精确的问题]
**子问题：** [2-5 个具体的未知领域]
**范围：** [地理、时间、领域边界]
**级别：** [快速扫描 / 深度研究 / 全面研究 / 持续情报]
**所需置信度：** [高 / 中 / 方向性]
**输出形式：** [报告 / 备忘录 / 简报 等]
**截止时间：** [时间约束]
```

**在开始执行之前，必须将协议展示给用户确认。**

## 第二阶段：证据收集

**目标：** 收集多样化、高质量、直接针对研究问题的来源。

### 指导原则

1. **工具是天生能力** — 按需使用 `WebSearch`、`QuickSearch`、`QuickExplore`、`FindRelation`、`WebFetch`。无需特殊说明。
2. **来源多样化** — 不要依赖单一类型来源。混合网络文章、学术论文、官方文件、分析师报告。
3. **记录来源信息** — 对每个来源记录：URL/DOI、发布者、作者、发布日期、获取日期。
4. **质量优于数量** — 5 个高质量来源胜过 50 篇博客文章。

### 来源质量评分

从 4 个维度对每个来源进行评分（1-5 分）：

| 维度 | 衡量内容 | 危险信号 |
|------|---------|---------|
| **可信度** | 发布机构权威性、方法论严谨性 | 匿名博客、未说明方法论、已知偏见 |
| **时效性** | 信息的更新程度 | 快速变化领域超过 2 年的数据 |
| **相关性** | 是否直接针对研究问题 | 仅附带提及 |
| **偏见判断** | 平衡视角还是宣传性质 | 供应商赞助的"研究"、无反方证据 |

**最低门槛：** 平均分 ≥ 3.5 才能纳入最终综合。低分来源也要标记但不能丢弃——它们可能提供有用的上下文。

### 收集检查清单

- [ ] 每个关键主张至少有 2 个独立来源
- [ ] 来源涵盖多种视角（看多、看空、中立）
- [ ] 优先使用一手来源而非二手解读
- [ ] 来源记录完整的引用元数据

## 第三阶段：多智能体编排

**目标：** 根据决策风险大小，按比例扩展研究投入。

### 何时扩展

| 研究级别 | 智能体数量 | 何时部署额外智能体 |
|---------|-----------|------------------|
| Level 1 — 快速扫描 | 1（仅主导者） | 从不 |
| Level 2 — 深度研究 | 1-2 | 如果来源涉及差异很大的领域（如技术+金融） |
| Level 3 — 全面研究 | 2-4 | 始终——并行工作流 |
| Level 4 — 持续情报 | 2-4 + 持续 | 始终 |

### 角色定义

| 角色 | 职责 | 触发条件 |
|------|------|---------|
| **主导研究员** | 问题设计、来源协调、综合、质量控制 | 始终 |
| **数据分析师** | 定量分析、统计工作、数据清洗、图表制作 | 当定量数据是主要组成部分时 |
| **领域专家** | 领域验证、方法论评审、合理性检查 | 需要特定领域专业知识时 |
| **撰稿人** | 报告起草、叙事结构、执行摘要 | 输出为正式报告时（全面研究+） |

使用 `team-create` 和 `task-create` / `task-update` 协调并行工作流。

## 第四阶段：分析与综合

**目标：** 将收集到的证据转化为洞察、模式和可操作的结论。

### 步骤 4.1 — 证据映射

针对协议中的每个子问题，映射证据：

```
子问题：[来自协议]

支持证据：
- [主张] → [来源引用]（置信度：高/中/低）
- [主张] → [来源引用]（置信度：高/中/低）

矛盾证据：
- [主张] → [来源引用]（置信度：高/中/低）

空白：
- [我们仍然不知道或证据不足的事项]
```

### 步骤 4.2 — 模式发现

关注：
- **趋同** — 多个独立来源得出相似结论（增加置信度）
- **分歧** — 来源之间存在分歧（记录分歧，不要平滑处理）
- **空白** — 证据不足的问题（标记为不确定性）
- **时间趋势** — 事物随时间的变化

### 步骤 4.3 — 假设形成

每个重大发现都要：

1. 清晰陈述假设
2. 确定支持证据（带来源引用）
3. 确定矛盾证据（主动寻找反证）
4. 根据证据强度分配置信度级别
5. 记录剩余的不确定性

### 步骤 4.4 — 事实与假设分离

**每个结论必须标记为以下之一：**

| 标签 | 含义 | 示例 |
|------|------|------|
| **事实** | 有 2 个以上独立、高质量来源直接支持 | "X 公司于 2025 年 3 月完成 5 亿美元 D 轮融资（来源 A、来源 B）" |
| **综合发现** | 由多个数据点推导得出；每个数据点是事实，但结论是解释性的 | "基于招聘动态和合作伙伴公告，X 公司正在拓展亚太市场（来源 C、来源 D）" |
| **假设** | 有合理依据但直接证据有限的推断；必须明确标注 | "假设当前增速持续，X 公司将在 18 个月内需要新一轮融资（基于来源 E 的推算）" |
| **推测** | 有依据的意见，但证据不充分；必须标明 | "X 公司可能成为大型科技公司的收购目标（无直接证据，根据市场整合趋势推断）" |

**绝不允许将假设或推测当作事实呈现。** 如果用户需要更高的置信度，指出哪些来源可以消除不确定性。

### 置信度级别

| 级别 | 标准 |
|------|------|
| **高** | 3 个以上独立、高质量来源支持；有定量数据；无重大矛盾 |
| **中** | 2 个以上来源支持但存在一定局限性；定性共识；存在轻微矛盾 |
| **低** | 单来源或证据有限；存在重大矛盾；主要是推测性的 |

## 第五阶段：报告产出

**目标：** 生成结构化的 Markdown 文件，保存到当前工作目录。

### 输出位置

将报告保存为当前工作目录下的 `.md` 文件，文件名清晰描述内容：

```
<项目目录>/research-<主题缩写>-<YYYY-MM-DD>.md
```

例如：`research-ai-healthcare-diagnostics-2025-06-24.md`

### 报告结构要求

```markdown
# 研究：[标题]

**日期：** YYYY-MM-DD
**研究级别：** 快速扫描 / 深度研究 / 全面研究 / 持续情报
**决策背景：** [这项研究服务于什么决策]

---

## 执行摘要

[一段话总结核心结论。3-5 个关键发现及置信度。如适用，附建议。]

---

## 1. 研究方法

- **核心问题：** [来自协议]
- **范围：** [地理、时间、领域边界]
- **来源：** [按类型统计数量、质量分布]
- **局限性：** [本研究未覆盖的内容]

## 2. 发现

[按主题组织，而非按来源。每个发现包括主张、支持证据（带引用）、置信度。]

## 3. 分析

[跨发现的综合、模式解读、影响分析。]

## 4. 结论

[直接回答核心问题和子问题。按事实/综合发现/假设/推测标注。]

## 5. 建议

[可操作、有优先级、附证据引用。]

---

## 附录

### A. 来源列表

| 编号 | 标题 | 发布者 | 日期 | 可信度 | 时效性 | 相关性 | 偏见 | 链接 |
|------|------|--------|------|--------|--------|--------|------|------|

### B. 不确定性与空白

[仍然未知的内容。需要什么才能解决。]

### C. 原始协议

[从第一阶段复制的研究协议，以便追溯。]
```

### 引用规则

1. **每个事实性主张必须有 ≥1 个来源引用** — 格式：`[来源 N]`
2. **相互矛盾的来源必须被承认** — 不要掩盖分歧
3. **区分**直接引用、转述和综合结论
4. **假设和推测必须标注** — 使用第四阶段 4.4 的标签

## 第六阶段：验证与反思

**目标：** 在交付输出之前，验证报告是否真正诚实地回答了最初的研究问题。

### 步骤 6.1 — 目标一致性检查

重新阅读原始研究协议，并检查：

- [ ] 报告是否直接回答了核心问题？（而不是另一个问题、也不是相近的问题）
- [ ] 是否涵盖了协议中的每个子问题？
- [ ] 所有结论是否在约定的范围内？（没有未经同意的范围蔓延）
- [ ] 如果决策锚定是"我是否应该投资 X"，报告是否给出了明确的答案或可操作的决策框架？

如果任何检查未通过，报告是不完整的——修改或明确标注空白。

### 步骤 6.2 — 证据完整性检查

- [ ] 每个事实性主张是否有至少一个来源引用？
- [ ] 每个重大发现是否分配了置信度级别？
- [ ] 假设和推测是否被明确标注？（不要"结论包装"——不要把假设写得像事实）
- [ ] 矛盾或不同意见的来源是否被承认？
- [ ] 是否存在没有证据支持的声明？（如果有，删除或标记为推测）

### 步骤 6.3 — 推理完整性检查

- [ ] 从证据到结论的推理链是否成立？还是存在逻辑跳跃？
- [ ] 所有数字类主张是否可追溯到其来源计算？（例如"市场将在 2030 年达到 X 美元"——哪位分析师？什么方法论？）
- [ ] 相关性是否与因果性区分开来？
- [ ] 如果报告做出了预测或展望，是否说明了其基础模型/假设？

### 步骤 6.4 — 最终置信度声明

以诚实评估结尾：

> **本报告置信度：** [高 / 中 / 低]
>
> **关键不确定性：** [如果出现新证据，哪些内容会改变？]
>
> **建议跟进：** [如果需要更高置信度，应该做什么——例如"委托一手调研"、"访谈 3 位行业高管"、"购买某报告"]

## 反模式

1. **确认偏差 / 挑拣数据**
   不要选择性地纳入支持已有结论的来源，而忽略矛盾证据。主动寻找反证。如果找不到，将这一缺失记录为局限性。

2. **回答了错误的问**
   最常见的失败模式。用户问的是"市场机会"，得到的却是"技术概述"。始终锚定在协议上。如果在研究过程中出现了新问题，将它们归类为"额外发现"，而不是让它们改变报告的方向。

3. **把假设当事实呈现**
   证据只支持"X 公司的招聘动态表明他们正在探索这个市场"，绝不要写成"X 公司将进入这个市场"。这个区别至关重要。必须标注清楚。

4. **混淆相关与因果**
   两个趋势同时出现，不等于一个导致了另一个。明确说明证据显示的是什么，以及你推断的是什么。

5. **忽略来源偏见**
   每个来源都有其立场。供应商的白皮书推广他们的解决方案。做空报告强调风险。学术论文可能存在资助方的利益冲突。描述并披露来源偏见。

6. **基于有限样本过度概**
   "我们分析了 5 家公司，发现......"不等于"整个行业都是......"。精确说明范围和样本量。外推必须始终标注。

7. **跳过问题定义阶段**
   不先定义研究问题就直接开始网络搜索，会导致范围蔓延、精力浪费和输出缺乏焦点。在 Phase 1 上投入总时间的 10-15%。

8. **产出报告后不做验证**
   跳过 Phase 6 意味着交付的报告可能根本没有回答最初的问题。在交出输出之前，始终运行验证检查。
